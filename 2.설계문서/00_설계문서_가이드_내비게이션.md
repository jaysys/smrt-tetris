# 설계문서 가이드 내비게이션

## 1. 문서 목적
이 문서는 `2.설계문서` 폴더에 들어온 개발자, 테스터, 운영 담당자가 어떤 문서를 어떤 순서로 읽어야 하는지 빠르게 이해하도록 돕는 시작점 문서이다.

본 폴더의 문서는 다음 목적을 충족해야 한다.

- 개발자가 문서를 보고 바로 개발 착수할 수 있어야 한다.
- 테스터가 기능 단위로 테스트를 수행하고 완료 여부를 판정할 수 있어야 한다.
- 프로젝트 참여자가 동일한 요구사항 해석 기준을 공유할 수 있어야 한다.

## 2. 권장 읽기 순서
처음 폴더에 들어온 경우 아래 순서로 문서를 확인한다.

1. `00_설계문서_가이드_내비게이션.md`
   - 전체 문서 구조와 읽기 순서를 이해한다.
2. `01_기능설계서.md`
   - 무엇을 만들어야 하는지 기능 요구사항과 처리 규칙을 파악한다.
3. `02_시스템설계서.md`
   - 어떤 구조와 방식으로 구현할지 아키텍처와 모듈 구성을 파악한다.
4. `03_API명세서.md`
   - 화면, 서버, 외부 연동 간 인터페이스 계약을 확인한다.
5. `04_DB설계서.md`
   - 저장 구조, 데이터 관계, 제약조건을 확인한다.
6. `05_화면흐름도_및_사용자시나리오.md`
   - 사용자 행동 흐름과 화면 전환을 확인한다.
7. `06_MVP개발계획서.md`
   - 기능별 우선순위와 구현 순서를 확인한다.
8. `07_테스트계획서.md`
   - 테스트 범위, 테스트 단위, 판정 기준, 완료조건을 확인한다.
9. `08_기능별_테스트수행_MVP.md`
   - 어떤 구현 단계에서 어떤 테스트를 바로 수행할 수 있는지 확인한다.
10. `09_요구사항추적표.md`
    - 요구사항과 설계, 구현, 테스트 간 추적 관계를 확인한다.
11. `10_운영배포체크리스트.md`
    - 배포 및 운영 준비 상태를 확인한다.
12. `11_용어사전_및_가정사항.md`
    - 용어 기준, 설계 전제, 범위 제외 항목을 확인한다.
13. `12_게임규칙_수치테이블.md`
    - 게임 엔진 구현과 단위 테스트를 위한 정량 규칙을 확인한다.
14. `13_테스트데이터_명세서.md`
    - 독립 테스트를 위한 계정, 시드 데이터, 초기화 절차를 확인한다.
15. `14_QA실행가이드.md`
    - 로컬 실행, 테스트 순서, 날짜 전환과 장애 재현 절차를 확인한다.
16. `15_API_예시응답_모음.md`
   - 성공/실패/중복/권한 오류 응답 예시를 확인한다.
17. `16_D0완료_설계자검토의견.md`
   - 개발팀의 D0 완료 선언에 대한 설계자 판단과 후속 우선순위를 확인한다.
18. `17_기획자검토의견_설계자회신.md`
   - 기획자의 설계 검토 의견에 대한 설계자의 수용 여부와 후속 반영 방침을 확인한다.
19. `18_개발자전달_설계변경사항.md`
   - 최근 설계 변경 중 개발팀이 즉시 반영해야 할 변경사항과 영향 범위를 확인한다.
20. `19_게임룰_외부참조기준.md`
   - 게임 화면 내 블록 동작 규칙이 어떤 외부 기준을 참조해 설계되었는지 확인한다.
21. `20_개발자전달_게임엔진구현지시서.md`
   - 개발자가 게임 엔진 구현 시 따라야 할 고정 규칙과 구현 순서를 확인한다.
22. `21_개발자전달_추가설계변경사항.md`
   - 마지막 개발자 전달 문서 이후 추가된 설계 변경 차분을 확인한다.
23. `22_설계자_개발자검토의견서_회신.md`
   - 개발자의 설계변경 검토 의견서에 대한 설계자 회신과 보완 요청을 확인한다.
24. `23_UI화면설계서.md`
   - 사용자 화면별 UI 구조, 모바일 세로 배치, CTA 위계를 확인한다.
25. `24_개발자전달_UI화면설계_요약.md`
   - UI 화면설계 변경을 개발자에게 바로 전달하기 위한 요약과 구현 우선순위를 확인한다.
26. `25_기획자_화면설계개선의견_설계자회신.md`
   - 기획자의 화면설계 개선의견에 대한 설계자 회신과 반영 결과를 확인한다.
27. `26_개발자전달_캐주얼게임_UI강화요청서.md`
   - 개발자에게 캐주얼 게임 서비스 요소를 반영하도록 요청하는 전달 문서를 확인한다.

## 3. 문서별 역할
### `01_기능설계서.md`
- 기능의 목적, 입력/출력, 처리 규칙, 예외, 권한, 상태 변화를 정의한다.
- 개발자가 업무 규칙을 오해하지 않도록 기준을 제공한다.

### `02_시스템설계서.md`
- 시스템 구조, 기술 아키텍처, 모듈 책임, 처리 흐름, 비기능 요구사항을 정의한다.
- 개발자가 실제 구현 방식을 결정할 수 있도록 한다.

### `03_API명세서.md`
- API 요청/응답 구조, 인증, 오류 응답, 필드 정의를 명시한다.
- 프론트엔드, 백엔드, 외부 연동 간 계약 문서 역할을 한다.

### `04_DB설계서.md`
- 테이블 구조, 컬럼 정의, 관계, 인덱스, 제약조건을 명시한다.
- 데이터 저장 방식과 정합성 기준을 제공한다.

### `05_화면흐름도_및_사용자시나리오.md`
- 사용자 역할별 주요 흐름과 화면 전환을 정리한다.
- 화면 단위 개발과 시나리오 테스트의 기준이 된다.

### `06_MVP개발계획서.md`
- 기능 우선순위, 선행 의존성, 개발 단계, MVP 범위를 정의한다.
- 개발팀이 어떤 순서로 구현해야 하는지 판단할 수 있게 한다.

### `07_테스트계획서.md`
- 테스트 대상, 세부 케이스, 전제조건, 수행절차, 기대결과, 완료조건을 정의한다.
- 테스터가 기능 단위로 수행 가능한 테스트 문서 역할을 한다.

### `08_기능별_테스트수행_MVP.md`
- 기능 개발 단계와 테스트 착수 시점을 연결한다.
- 개발 완료 직후 어떤 테스트를 바로 실행해야 하는지 보여준다.

### `09_요구사항추적표.md`
- 기획 요구사항이 어떤 설계와 테스트로 연결되는지 추적한다.
- 누락 기능이나 미검증 요구사항을 발견하기 쉽게 한다.

### `10_운영배포체크리스트.md`
- 배포 전 점검, 환경설정, 운영 준비, 장애 대응 기준을 정리한다.
- 릴리즈 직전 확인 문서 역할을 한다.

### `11_용어사전_및_가정사항.md`
- 용어 정의, 설계 가정, 미확정 이슈, 범위 제외 항목을 정리한다.
- 해석 차이로 인한 구현 오류를 줄인다.

### `12_게임규칙_수치테이블.md`
- 회전, 낙하 속도, 락 딜레이, 입력 반복, 판정 순서를 수치로 고정한다.
- 게임 엔진과 테스트 코드의 기준 문서가 된다.

### `13_테스트데이터_명세서.md`
- 테스트용 사용자, 운영자, 랭킹 샘플, 챌린지 샘플, 금칙어 샘플을 정리한다.
- 제3자 독립 테스트의 재현 기준이 된다.

### `14_QA실행가이드.md`
- 로컬 실행, 테스트 URL, DB 초기화, 날짜 전환, 네트워크 장애 테스트 절차를 정리한다.
- 외부 테스터가 개발자 도움 없이 환경을 준비할 수 있게 한다.

### `15_API_예시응답_모음.md`
- API 성공/실패/검증 실패/권한 오류/중복 요청 예시를 정리한다.
- 프런트엔드, 백엔드, QA 간 응답 해석 차이를 줄인다.

### `16_D0완료_설계자검토의견.md`
- 개발팀의 D0 완료 선언을 설계 기준으로 검토한 의견을 정리한다.
- 현재 단계 판단과 D1 우선순위를 전달하는 문서 역할을 한다.

### `17_기획자검토의견_설계자회신.md`
- 기획자의 설계 검토 의견별 수용 여부를 정리한다.
- 즉시 반영 사항과 후속 범위 구분 기준을 전달하는 문서 역할을 한다.

### `18_개발자전달_설계변경사항.md`
- 최근 설계 변경사항 중 개발 구현에 직접 영향을 주는 항목을 요약한다.
- 개발자가 코드, 테스트, 일정 우선순위를 즉시 조정할 수 있게 하는 전달 문서 역할을 한다.

### `19_게임룰_외부참조기준.md`
- 게임 화면 내 블록 생성, 이동, 회전, Hold, Ghost 관련 규칙의 외부 참조 기준을 정리한다.
- 공개 출처 기반 규칙과 프로젝트 고유 튜닝값을 구분하는 문서 역할을 한다.

### `20_개발자전달_게임엔진구현지시서.md`
- 개발자에게 전달할 게임 엔진 구현 기준, 금지 해석, 최소 테스트 요구를 정리한다.
- 구현 과정에서 화면 내 블록 동작 규칙이 흔들리지 않도록 하는 인계 문서 역할을 한다.

### `21_개발자전달_추가설계변경사항.md`
- 마지막 개발자 전달 문서 이후의 추가 설계 변경만 추려 전달한다.
- 개발자가 기존 전달사항과 새 엔진 규칙 변경을 구분해 반영할 수 있게 하는 차분 문서 역할을 한다.

### `22_설계자_개발자검토의견서_회신.md`
- 개발자의 설계변경 검토 의견서에 대한 설계자 판단과 보완 요청을 정리한다.
- 개발팀 내부 문서와 설계 기준 사이의 해석 차이를 줄이는 회신 문서 역할을 한다.

### `23_UI화면설계서.md`
- 사용자 화면별 UI 구조, 정보 위계, 모바일 세로 배치, 반응형 전환 규칙을 정리한다.
- 개발자가 임의 화면 구성을 하지 않고 기획 의도에 맞는 화면을 구현하게 하는 전용 화면설계 문서 역할을 한다.

### `24_개발자전달_UI화면설계_요약.md`
- UI 화면설계서의 핵심 변경과 구현 우선순위를 짧게 요약한다.
- 개발자가 화면설계서를 전달받고 바로 착수할 수 있게 하는 인계 문서 역할을 한다.

### `25_기획자_화면설계개선의견_설계자회신.md`
- 기획자의 화면설계 상향 요구에 대한 수용 여부와 반영 결과를 정리한다.
- UI 설계 기준이 어떤 방향으로 강화되었는지 확인하는 회신 문서 역할을 한다.

### `26_개발자전달_캐주얼게임_UI강화요청서.md`
- 개발자에게 캐주얼 게임 서비스 요소 반영을 명시적으로 요청하는 전달 문서다.
- Hero, 보상 언어, 경쟁 무드, 게임다운 조작부를 구현 기준으로 고정하는 역할을 한다.

## 5. 개발 착수 기준
개발은 아래 조건이 충족되면 착수 가능 상태로 본다.

- 기능설계서에 기능 범위와 예외 처리 기준이 정의되어 있다.
- 시스템설계서에 구현 구조와 모듈 책임이 정의되어 있다.
- API명세서와 DB설계서에 인터페이스와 데이터 구조가 정의되어 있다.
- MVP개발계획서에 개발 순서와 선행조건이 정리되어 있다.

## 6. 테스트 착수 기준
테스트는 아래 조건이 충족되면 수행 가능 상태로 본다.

- 테스트계획서에 테스트 단위와 수행 절차가 정의되어 있다.
- 각 테스트 항목마다 기대결과와 완료조건이 명시되어 있다.
- 기능별 테스트수행 MVP 문서에 테스트 가능한 시점이 연결되어 있다.
- 요구사항추적표로 테스트 누락 여부를 확인할 수 있다.
- 테스트데이터 명세서와 QA실행가이드로 동일 환경 재현이 가능하다.

## 7. 문서 작성 원칙
- 모든 문서는 개발자와 테스터가 즉시 실행 가능한 수준으로 구체적으로 작성한다.
- 완료조건은 정성적 표현이 아니라 검증 가능한 문장으로 작성한다.
- 모호한 표현이 있으면 용어사전 또는 가정사항 문서에 기준을 남긴다.
- 기능, 설계, 테스트 문서는 서로 참조 가능하도록 동일한 기능 ID 또는 요구사항 ID를 사용한다.

## 8. 폴더 운영 원칙
- 문서 번호는 읽기 순서와 참조 우선순위를 반영한다.
- 문서가 추가되더라도 본 내비게이션 문서를 먼저 갱신한다.
- 세부 설계가 변경되면 기능설계서, 시스템설계서, 테스트계획서, 요구사항추적표를 함께 점검한다.

## 9. 현재 상태
현재 본 문서는 설계문서 폴더의 기준 구조를 정의한 초기 가이드이다.
향후 `1.기획서`의 요구사항 분석이 완료되면 각 상세 문서를 이 구조에 맞춰 생성한다.
