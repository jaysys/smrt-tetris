# 테스터 전달 자동화대응 검토의견

## 1. 목적
본 문서는 `4.테스터/03_테스터_요구사항_자동화대응.md`에 대한 개발팀 검토 결과와 반영 방향, 추가 협의가 필요한 사항을 정리한다.

## 2. 검토 결론
- 전체 방향은 타당하다.
- 본 요구사항은 선택 보강이 아니라 개발관리 기준으로 수용한다.
- 개발계획서에는 자동화 대응 원칙, 단계별 자동화 산출물, 테스트 전용 제어 기준을 반영했다.
- 다만 일부 항목은 구현 기준을 더 명확히 하기 위해 범위와 표현을 조정할 필요가 있다.

## 3. 수용 항목
### 3.1 Must 요구사항
- `TQA-001` 웹 흐름 헤드리스 E2E 지원: 수용
- `TQA-002` 핵심 요소 `data-testid` 부여: 수용
- `TQA-003` 서버 주입 시계 `TEST_CLOCK_ISO_UTC` 지원: 수용
- `TQA-004` 결과 제출/보상 수령/운영 설정 반영의 초기화 절차 제공: 수용
- `TQA-005` 결과 제출 멱등성 재현 가능성: 수용
- `TQA-006` 운영자/게스트/시드 데이터 재생성 스크립트: 수용
- `TQA-007` 실패 산출물 수집: 수용
- `TQA-008` `pnpm test:e2e` 단일 명령 제공: 수용

### 3.2 Should 요구사항
- `TQA-101` 시드 고정과 입력 시퀀스 재생: 수용
- `TQA-102` 운영 기능 API/UI 이중 검증: 수용
- `TQA-103` 랭킹/미션/업적 테스트 데이터 팩: 수용
- `TQA-104` 모바일 뷰포트 기본 해상도 제공: 수용

## 4. 조정 또는 명확화가 필요한 항목
### 4.1 응답 검증 필드 범위 조정
- 문서 5.2의 `success`, `error.code`, `data.validationStatus`, `data.personalBest`, `data.leaderboardUpdated`, `meta.idempotentReplay`, `meta.retryable`는 모든 핵심 API 공통 필드로 보기 어렵다.
- `data.validationStatus`, `data.personalBest`, `data.leaderboardUpdated`, `meta.idempotentReplay`, `meta.retryable`는 결과 제출 API 중심 검증 필드로 한정하는 것이 맞다.
- 요청사항: 엔드포인트별 필수 검증 필드를 구분해 재정리해 달라.

### 4.2 테스트 전용 제어 수단의 표준 경로 필요
- 테스트 DB 초기화, 시드 데이터 재적재, 테스트 시계 주입, 테스트 관리자 접근은 필요하다.
- 다만 구현 시 혼선을 줄이기 위해 `TEST_MODE` 환경에서만 열리는 표준 인터페이스가 필요하다.
- 개발팀 제안:
  - 명령형: `pnpm db:reset`, `pnpm db:seed`, `pnpm test:e2e`
  - API형: `/v1/test/*` 또는 동등한 테스트 전용 엔드포인트

### 4.3 실패 산출물 저장 규칙 필요
- 스크린샷, 응답, 콘솔 로그, 네트워크 로그 수집 요구는 타당하다.
- 다만 저장 경로와 파일명 규칙이 없으면 CI와 로컬 결과 수집이 일관되지 않는다.
- 요청사항: 최소 저장 경로 규약과 보존 범위를 정의해 달라.

### 4.4 게임 자동화 지원 방식 우선순위 결정 필요
- 문서 4.3의 A/B/C 모두 타당하나, MVP에서는 우선순위가 필요하다.
- 개발팀 제안 우선순위:
  1. 시드 고정 + 입력 재생
  2. 테스트 전용 디버그 훅
  3. 테스트 전용 상태 주입 API
- 이유: 엔진 재현성과 운영 환경 분리를 동시에 만족하기 쉽기 때문이다.

### 4.5 `pnpm install` 인수 기준의 환경 전제 명시 필요
- 현재 워크스페이스는 `registry.npmjs.org` DNS 제한으로 설치 검증이 차단된 상태다.
- 따라서 `pnpm install` 인수 기준은 네트워크 접근 가능 환경 또는 사내 패키지 미러 환경을 전제로 명시해야 한다.

## 5. 개발팀 반영 내용
- `3.개발자/07_개발관리_개발계획서.md`에 자동화 대응 원칙을 반영했다.
- D0~D6 단계별로 자동화 관점의 산출물과 통과 조건을 추가했다.
- `Playwright`, `Vitest`, 테스트 전용 게이트, 실패 산출물 수집, 테스트 명령 기준을 개발관리 기준에 포함했다.

## 6. 요청 사항
- 테스터 문서에 엔드포인트별 필수 검증 필드 구분을 추가해 달라.
- 실패 산출물 저장 규약을 정의해 달라.
- 게임 자동화 지원 방식의 우선순위에 동의하는지 회신해 달라.
- `pnpm install` 인수 기준의 네트워크 전제를 명시해 달라.

## 7. 결론
- 요구사항의 핵심 취지는 전부 수용한다.
- 현재 미조정 항목은 반려 사유가 아니라 구현 기준을 닫기 위한 명확화 요청이다.
- 위 4개 조정 사항이 정리되면 자동화 요구사항을 개발 기준으로 안정적으로 운영할 수 있다.
