# 테스터 전달 D0 재시험 요청

## 1. 목적
- 본 문서는 `4.테스터/06_D0_테스트결과서.md`의 실패 항목을 개발 측에서 재확인한 결과와 재시험 요청 사항을 정리해 전달하기 위한 문서다.

## 2. 재확인 결과 요약
| 항목 | 테스터 결과서 | 개발 측 재확인 | 판단 |
|---|---|---|---|
| `pnpm db:seed` | 1차 Fail, 재테스트 Pass | `pnpm db:reseed` 기준 Pass | 추가 실패 아님 |
| `GET /v1/test/ping` | Fail `404` | `pnpm start:api:test` 기준 Pass | 재시험 필요 |
| `POST /v1/test/reset` | Fail `404` | `pnpm start:api:test` 기준 Pass | 재시험 필요 |

## 3. 판단
- `db:seed`는 최신 결과서 기준으로 이미 재테스트 Pass가 확인되었으므로 D0 잔여 실패 항목에서 제외한다.
- 현재 남은 확인 대상은 `TEST_MODE` 전용 엔드포인트 2건뿐이다.
- 개발 측 재확인에서는 아래 절차로 정상 응답을 확인했다.

## 4. 재시험 표준 절차
### 4.1 API 기동
```bash
pnpm start:api:test
```

기대 조건:
- `TEST_MODE=true`
- `HOST=127.0.0.1`
- `PORT=3001`
- 로그에서 `/v1/test/ping`, `/v1/test/reset` 라우트 매핑 확인 가능

### 4.2 호출 대상
```bash
curl -sf http://127.0.0.1:3001/v1/test/ping
curl -sf -X POST http://127.0.0.1:3001/v1/test/reset
```

### 4.3 기대 결과
- `GET /v1/test/ping`
  - `success=true`
  - `data.status="ok"`
  - `data.testMode=true`
- `POST /v1/test/reset`
  - `success=true`
  - `data.accepted=true`

## 5. 재시험 요청 범위
- D0 전체를 다시 수행할 필요는 없다.
- 아래 2건만 재시험 후 결과서를 갱신해 주면 된다.
  - `GET /v1/test/ping`
  - `POST /v1/test/reset`

## 6. 개발 측 반영 사항
- 루트 표준 명령 추가:
  - `pnpm start:api:test`
  - `pnpm db:reseed`
- 관련 관리 문서 반영 완료:
  - `3.개발자/07_개발관리_개발계획서.md`
  - `3.개발자/08_작업관리대장.md`
  - `3.개발자/09_이슈_결함대장.md`
  - `3.개발자/10_의사결정기록서.md`

## 7. 요청 사항
- 위 표준 절차로 `TEST_MODE` 엔드포인트 2건을 재시험해 달라.
- Pass 확인 시 `4.테스터/06_D0_테스트결과서.md`와 필요 시 `4.테스터/05_현재_테스트가능범위_식별서.md`를 갱신해 달라.
