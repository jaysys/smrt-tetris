# 시스템설계서

## 1. 문서 개요
- 문서명: 테트리스형 웹서비스 시스템설계서
- 목적: 개발팀이 실제 구현을 시작할 수 있도록 기술 아키텍처, 모듈 구조, 처리 시퀀스, 비기능 요구사항을 정의한다.

## 2. 설계 목표
- 로그인 없이 즉시 플레이 가능한 웹서비스
- 모바일/데스크톱 공통 대응
- 기록과 랭킹 검증이 가능한 서버 구조
- 운영자가 데일리 챌린지와 공지를 조정할 수 있는 백오피스
- 부정행위 탐지와 랭킹 오염 방지 구조

## 2.1 시간 기준 표준
- DB 저장 시간은 전부 UTC로 저장한다.
- API 응답의 시간 필드는 UTC ISO 8601 문자열로 반환한다.
- 프런트엔드 표시 시간은 기본 KST(UTC+09:00) 변환을 사용한다.
- Daily Challenge 활성화 기준은 KST 00:00:00이다.
- 일간 랭킹 period 산정은 KST 기준 날짜 경계다.
- 주간 랭킹 시작은 `월요일 KST 00:00:00`이다.

## 6. 핵심 설계 결정
### 6.1 게임 로직 위치
- 실시간 입력과 렌더링은 클라이언트에서 처리한다.
- 최종 결과 반영은 서버 검증을 거친다.
- 서버는 `세션 발급 시 seed`, `모드 설정`, `시작 시각`, `허용 시간`, `검증용 체크포인트 규칙`을 내려준다.

### 6.2 부정행위 방지 구조
- 세션 시작 시 `server_session_id`, `seed`, `issued_at`, `mode_config_version`을 발급한다.
- 클라이언트는 결과 제출 시 `종료 시각`, `입력 요약`, `라인 제거 수`, `점수`, `체크포인트 해시`를 전송한다.
- 서버는 모드 규칙과 시간 범위, 입력 빈도, 점수 계산 일관성을 점검한다.
- 검증 실패 시 결과는 `FLAGGED` 상태로 저장하고 랭킹 반영을 차단한다.

### 6.2.1 부정행위 검증 정량 기준
| 항목 | ACCEPTED | FLAGGED | REJECTED |
|---|---|---|---|
| 체크포인트 해시 | 전부 일치 | 부분 누락 1건 이하, 재전송 필요 | 불일치 또는 위조 흔적 |
| Sprint 40라인 기록 | 90초 이상 | 60초 이상 90초 미만 | 60초 미만 |
| Marathon 점수 증가율 | 규칙상 가능한 범위 | 이론 최대치 95% 초과 | 이론 최대치 초과 |
| 입력 빈도 | 초당 12회 이하 | 초당 12회 초과 20회 이하 | 초당 20회 초과 |
| 결과 제출 중복 | 동일 key 1회 저장 | 동일 결과 재전송 | 다른 payload로 동일 key 재사용 |

추가 규칙
- `FLAGGED` 결과는 `leaderboard_entries` 생성 금지
- `REJECTED` 결과는 `game_results`에 무효 사유를 남기고 공식 기록 반영을 차단
- 운영자 수동 승인 시 `FLAGGED -> ACCEPTED`
- 운영자 수동 확정 시 `FLAGGED -> REJECTED`

### 6.3 랭킹 생성 방식
- MVP에서는 SQLite3 원본 데이터를 기준으로 랭킹을 조회한다.
- 초기 트래픽 규모에서는 별도 Redis 캐시 없이도 운영 가능하다.
- 랭킹 갱신은 결과 승인 직후 동기 반영하고, 주간/전체 재계산은 애플리케이션 스케줄러 작업으로 보정한다.
- 트래픽 증가로 랭킹 조회 부하가 커질 때만 인메모리 캐시 또는 Redis를 추가한다.

### 6.4 운영 설정 반영 방식
- 운영 설정은 `config_version`으로 관리한다.
- 챌린지, 공지, 시즌, 보상 수치 모두 유효 시작 시각과 종료 시각을 가진다.
- 게임 세션 시작 시점의 설정 버전을 결과 검증에도 동일 적용한다.

## 7. 처리 시퀀스
## 7.1 첫 방문 및 즉시 플레이
1. 클라이언트가 `GET /v1/bootstrap` 호출
2. 서버가 게스트 토큰, 오늘의 챌린지, 공지, 설정 기본값 반환
3. 사용자가 `바로 시작` 클릭
4. 클라이언트가 `POST /v1/game-sessions` 호출
5. 서버가 세션 ID, seed, 모드 규칙, 시작 제한 시간 반환
6. 클라이언트가 게임 화면 렌더링 후 플레이 시작

## 7.2 게임 종료 및 결과 제출
1. 클라이언트가 로컬에서 결과 계산 완료
2. `POST /v1/game-sessions/{id}/finish` 호출
3. 서버가 검증 로직 수행
4. 검증 성공 시 `game_results`, `leaderboard_entries`, `mission_progress`, `achievement_progress` 갱신
5. 응답으로 결과 승인 여부, 랭킹 갱신 여부, 보상 요약 반환

## 7.3 Daily Challenge 갱신
1. 운영자가 백오피스에서 다음 날 챌린지 저장
2. 스케줄러가 자정 시점 활성화
3. `bootstrap` 응답에 새 챌린지 정보 포함
4. 진행 중인 이전 날짜 챌린지는 완료 처리 후 읽기 전용 전환

## 7.4 비정상 기록 처리
1. 결과 제출
2. 서버 검증 실패 또는 의심 패턴 감지
3. `fraud_flags` 저장
4. 랭킹 제외
5. 운영자 검토 큐 노출

## 8. 브라우저 및 클라이언트 정책
### 8.1 반응형 정책
- 데스크톱: 캔버스 중심 2열 레이아웃
- 모바일 세로: 캔버스 상단, HUD 좌우 축소, 조작 버튼 하단 고정
- 최소 지원 폭: 360px

### 8.2 포커스 정책
- 탭 이탈 시 자동 일시정지
- 30초 초과 이탈 후 복귀 시 해당 세션은 `abandoned` 처리 가능
- Sprint에서 장시간 중단은 기록 제출 불가 처리

### 8.3 저장 정책
- 로컬 스토리지: 게스트 식별자, 설정, 최근 모드
- 서버 저장: 공식 기록, 업적, 챌린지 진행도, 제재 이력

## 9. API/서비스 경계
- 클라이언트는 점수 판정 로직을 가지되, 최종 공식 기록 권한은 서버만 가진다.
- 운영 백오피스는 동일 API를 재사용하되 `admin` 스코프가 필요하다.
- 분석 이벤트는 게임 UX를 지연시키지 않도록 비동기 전송한다.

## 9.1 멱등성과 중복 방지
- `POST /v1/game-sessions/{id}/finish`는 `Idempotency-Key` 헤더를 필수로 사용한다.
- 동일한 `sessionId + idempotency key + payload hash` 조합은 한 번만 공식 처리한다.
- 네트워크 재시도 시 같은 key와 같은 payload는 이전 응답을 재반환한다.
- 같은 key로 다른 payload가 오면 `409 CONFLICT`로 거절한다.

## 10. 배치 및 스케줄러
| 작업명 | 주기 | 목적 |
|---|---|---|
| Daily Challenge Activate | 매일 00:00 | 새 챌린지 활성화 |
| Weekly Ranking Rotate | 매주 월요일 00:00 | 주간 랭킹 초기화 |
| Fraud Recheck | 10분 | 의심 기록 재검증 |
| Leaderboard Rebuild | 1시간 | 랭킹 집계 보정 |
| Dormant Player Segment | 1일 | 복귀 대상 사용자 추출 |

## 10.1 복귀 사용자 경험 방향성
- `Dormant Player Segment`는 데이터 추출만으로 끝내지 않는다.
- 후속 범위에서 복귀 사용자 경험은 아래 4개 방향 중 하나 이상으로 확장할 수 있게 설계한다.
  - 복귀 배너
  - 복귀 미션
  - 최근 최고 기록 회상 메시지
  - Daily Challenge 재진입 유도
- MVP에서는 게스트 우선 구조를 유지하므로, 본 항목은 즉시 구현 대상이 아니라 후속 설계 방향성으로 관리한다.

## 11. 비기능 요구사항
### 성능
- `GET /v1/bootstrap` p95 300ms 이하
- `POST /v1/game-sessions` p95 200ms 이하
- `POST /finish` p95 500ms 이하
- 게임 렌더 루프 60fps 목표

### 가용성
- 웹 서비스 목표 가용성 99.5%
- 랭킹 조회 장애 시 개인 기록과 로컬 결과 화면은 유지

### 보안
- 모든 API는 HTTPS 강제
- 운영자 계정 2단계 인증 권장
- 결과 제출 API에 요청 서명과 레이트 리밋 적용

### 접근성
- 설정 변경은 키보드와 터치 모두 접근 가능
- 중요 상태 변화는 색상 외 텍스트/아이콘/애니메이션으로 동시 제공

### 분석
- 핵심 퍼널 이벤트는 코어 플레이 루프와 함께 수집을 시작한다.
- 최소 수집 이벤트는 `landing_view`, `quick_start_click`, `game_start`, `game_finish`, `retry_click`이다.

## 12. 장애 대응 원칙
- 캐시 미사용 기본 구조이므로 SQLite3 직접 조회를 기본 경로로 사용
- 추후 캐시 도입 시 캐시 장애 발생 시 DB 조회 fallback
- 분석 시스템 장애 시 게임 기능은 정상 동작
- 챌린지 설정 누락 시 안전 기본값으로 이전 날짜 챌린지를 재사용하지 않고 `오늘의 자유 플레이 미션`으로 대체

## 13. 개발 분리 원칙
- `game-engine`은 UI 프레임워크에 의존하지 않는 순수 함수 중심 모듈로 구현한다.
- 점수 계산, 충돌 판정, 라인 제거는 테스트 가능한 독립 단위여야 한다.
- API DTO와 DB 엔터티는 직접 공유하지 않고 `shared-types`에서 계약 타입을 관리한다.

## 14. 미해결 기술 이슈
| 이슈 | 내용 | 대응 |
|---|---|---|
| TI-01 | 공유 이미지 서버 생성 vs 클라이언트 렌더 | MVP는 클라이언트 우선, 서버 렌더는 2차 |

| TI-04 | 모바일 좌우 손잡이 조작 레이아웃 | MVP 제외, 입력 확장 구조 위에 2차 검토 |

## 15. SQLite3 적용 기준
- SQLite3는 MVP 단일 서비스 배포 기준에서 충분한 선택이다.
- DB 파일은 서버 로컬 디스크 또는 영속 볼륨에 저장한다.
- WAL 모드를 활성화해 읽기/쓰기 동시성을 확보한다.
- 대량 동시 접속, 다중 API 인스턴스, 고빈도 랭킹 조회가 발생하면 PostgreSQL 전환을 검토한다.
- 운영 백업은 정기 파일 백업과 배포 전 스냅샷 방식으로 수행한다.

## 16. 캐시/큐 필요성 판단
- `캐시`는 반드시 필요하지 않다.
- 현재 MVP 범위에서는 SQLite3 조회와 애플리케이션 메모리 캐시만으로도 충분하다.
- `큐`도 반드시 필요하지 않다.
- Daily Challenge 활성화, 주간 랭킹 초기화, Fraud 재검사는 프로세스 내부 스케줄러로 먼저 구현할 수 있다.
- 별도 큐는 아래 조건이 생길 때 도입한다.
  - 공유 이미지 서버 렌더링이 비동기 작업으로 커질 때
  - 분석 이벤트 적재량이 급증할 때
  - 부정행위 재검증 작업이 오래 걸릴 때
  - 다중 인스턴스 환경에서 작업 중복 제어가 필요할 때
