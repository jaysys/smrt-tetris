# 설계자 전달용 추가 보완 요청

## 1. 목적
본 문서는 `2.설계문서` 재검토 결과를 바탕으로, 설계자가 추가로 보완해야 하는 항목을 문서별 작업 요청 형태로 정리한 전달 문서다.

## 2. 전달 요약
현재 설계문서는 MVP 개발 착수 가능한 수준까지 상당 부분 보강되었다. 다만 아래 항목은 구현 충돌과 테스트 재현성 저하를 막기 위해 추가 반영이 필요하다.

우선 보완이 필요한 핵심 영역은 다음 4가지다.
- DB 기술 기준과 DB설계서 타입 체계 정합성 확보
- 게임 엔진 규칙의 절대 기준 확정
- 비핵심 API까지 포함한 계약 문서 완결
- 날짜 전환 테스트의 실제 수행 방법 명문화

## 3. 문서별 추가 보완 요청
### 3.1 `04_DB설계서.md`
현재 문제:
- 시스템설계서는 SQLite3를 기준으로 서술되어 있으나 DB설계서는 PostgreSQL 계열 타입 표현이 남아 있다.
- `uuid`, `jsonb`, `timestamptz`, `bigserial` 등은 SQLite 기준 구현 문서로 보기 어렵다.

요청 사항:
- SQLite MVP 기준 실제 저장 타입으로 재정의해 달라.
- UUID 생성 전략을 명시해 달라.
- JSON 저장 컬럼의 저장 방식과 직렬화 기준을 명시해 달라.
- 시간 컬럼 저장 형식과 애플리케이션 변환 규칙을 명시해 달라.
- 현재 MVP 기준과 향후 PostgreSQL 전환 시 변경 예정 항목을 구분해 달라.

수정 완료 기준:
- 구현자가 DB 마이그레이션을 작성할 때 문서 해석 없이 바로 스키마를 만들 수 있어야 한다.
- 테스터가 DB 상태를 점검할 때 문서와 실제 저장 구조가 일치해야 한다.

### 3.2 `12_게임규칙_수치테이블.md`
현재 문제:
- 보드 크기, 속도, 점수 규칙은 정리되었으나 블록 단위 절대 규칙이 부족하다.
- 현재 표현만으로는 엔진 구현 결과가 개발자마다 달라질 수 있다.

요청 사항:
- 블록별 초기 좌표를 절대값으로 명시해 달라.
- 블록별 초기 회전 상태를 명시해 달라.
- 각 블록의 셀 매트릭스 또는 좌표 집합을 정의해 달라.
- 시계 방향, 반시계 방향 회전 전이표를 추가해 달라.
- 일반 블록, I 블록, O 블록의 벽차기 규칙을 분리해 달라.
- 스폰 실패 판정 예시를 1개 이상 추가해 달라.

수정 완료 기준:
- 프런트엔드 게임 엔진과 서버 검증 로직이 동일 규칙을 재현할 수 있어야 한다.
- 회전, 스폰 실패, 벽차기 관련 테스트 케이스를 문서만으로 작성 가능해야 한다.

### 3.3 `03_API명세서.md`
현재 문제:
- 핵심 API는 상세하지만 나머지 엔드포인트는 필드 수준 계약이 부족하다.
- 화면 구현과 QA 기준이 엔드포인트별로 비대칭적이다.

우선 보완 대상:
- `GET /me/stats`
- `GET /settings`
- `GET /daily-challenge`
- `GET /missions`
- `GET /share-links/{shareId}`
- 운영 API 전체

요청 사항:
- 모든 엔드포인트에 요청/응답 필드 표를 추가해 달라.
- 각 필드의 Required, Nullable, enum, 길이 제약을 표시해 달라.
- 중첩 객체와 배열 항목 구조를 명시해 달라.
- 운영 API의 성공/실패 응답 예시를 추가해 달라.
- 가능하면 각 API별 대표 오류 케이스를 1개 이상 명시해 달라.

수정 완료 기준:
- 프런트엔드가 응답 구조를 추정하지 않고 구현 가능해야 한다.
- QA가 필드 누락과 계약 위반을 사양 기준으로 판정할 수 있어야 한다.

### 3.4 `14_QA실행가이드.md`
현재 문제:
- 날짜 전환 테스트 절차는 있으나 시간을 실제로 어떻게 제어할지 명시되지 않았다.
- 외부 테스터가 재현 가능한 수준으로는 아직 닫혀 있지 않다.

요청 사항:
- 시간 제어 방식을 1개 표준으로 확정해 달라.
- 예: 테스트 플래그, 서버 주입 시계, mock clock, 전용 관리자 기능 중 하나
- 날짜 전환 후 무엇을 어디서 확인하는지 명시해 달라.
- 예: `GET /v1/daily-challenge`, `GET /v1/leaderboards`, 관리자 화면, DB 조회 중 표준 경로 지정
- 로컬 QA와 자동화 QA가 같은 기준을 쓸 수 있도록 방법을 통일해 달라.

수정 완료 기준:
- 외부 테스터가 개발자 지원 없이 날짜 전환 테스트를 재현 가능해야 한다.
- 재현 절차와 확인 지점이 문서상 명확해야 한다.

### 3.5 `10_운영배포체크리스트.md`
현재 문제:
- 표준 명령은 정리되었으나 실제 저장소 스크립트와의 정합성은 아직 불명확하다.
- 현재 워크스페이스에서는 `package.json` 또는 workspace 설정 파일이 확인되지 않아 문서상 명령 존재 여부를 검증할 수 없었다.

요청 사항:
- 문서에 적은 명령이 실제 저장소 스크립트와 일치하는지 확인해 달라.
- 존재하지 않는 명령은 제거하거나 `예정 명령`으로 구분해 달라.
- 환경별 차이가 있으면 로컬/배포용 명령을 분리해 달라.
- 롤백 명령은 예시인지 확정 명령인지 표시해 달라.

수정 완료 기준:
- 신규 개발자 또는 테스터가 문서 명령만 따라도 실패하지 않아야 한다.
- 운영 절차 문서와 실제 실행 스크립트가 충돌하지 않아야 한다.

## 5. 전달 결론
현재 설계문서는 초기 대비 충분히 개선되었고, MVP 개발 착수는 가능하다. 다만 위 추가 보완이 반영되면 구현 편차와 QA 해석 차이를 더 줄일 수 있다.

설계자 보완 완료 후에는 다음 기준으로 다시 확인하면 된다.
- DB 문서와 실제 기술 선택이 일치하는가
- 게임 엔진 규칙이 절대값 기준으로 닫혀 있는가
- 모든 주요 API가 필드 수준 계약을 갖는가
- 날짜 전환 테스트가 문서만으로 재현 가능한가
