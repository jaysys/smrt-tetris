# 설계자 개발자검토의견서 회신

## 1. 목적
- 본 문서는 `3.개발자/19_설계변경_검토의견서.md`에 대한 설계자 회신서다.
- 목적은 개발자의 검토 방향을 수용하되, 설계 기준상 더 엄밀하게 보완해야 할 항목을 명시하는 것이다.

- `3.개발자/19_설계변경_검토의견서.md`

## 3. 총평
- 개발자의 전체 판단 방향은 타당하다.
- 특히 아래 결론은 설계자 입장에서도 수용 가능하다.
  - D1을 재오픈하지 않는다.
  - D2 착수 전 선행 보정 작업을 등록한다.
  - 엔진 규칙, 분석 이벤트, 모바일 세로, 접근성 최소 옵션을 후순위로 미루지 않는다.
- 다만 현재 의견서는 실무 우선순위 판단은 적절하지만, 일부 항목에서 `근거 범위`와 `보정 범위`를 더 엄밀하게 적어야 한다.

## 4. 항목별 회신
### 4.1 분석 이벤트 미충족 판단
판단:
- `수용`, 단 근거 표현 보완 필요

회신:
- 분석 이벤트 5종이 아직 구현 코드에 연결되지 않았다는 판단 자체는 타당하다.
- 다만 `특정 파일에 연결 코드가 없다`는 근거만으로는 부족하다.
- 설계 기준상 이 항목은 `공개 웹 전체 구현 코드에서 이벤트 이름 검색 시 확인되지 않는다`는 수준으로 근거를 넓혀 적는 편이 정확하다.

설계자 보완 요청:
- `apps/web/src/app/game-client.tsx` 단일 파일 부재가 아니라 `앱 코드 전체 미연결`로 표현을 수정하는 것이 바람직하다.
- 이후 작업지시도 `특정 컴포넌트 수정`이 아니라 `퍼널 전체 이벤트 연결` 기준으로 잡아야 한다.

### 4.2 엔진 보정 범위
판단:
- `수용`, 단 액션 범위 확장 필요

회신:
- 전이별 SRS 벽차기, 회전 테스트 순서, 점수 체계, 중력 규칙 미충족 판단은 타당하다.
- 다만 현재 의견서는 `Perfect Clear / Back-to-Back / Combo 점수 체계`와 `중력/소프트 드롭`만 액션으로 적혀 있어, 설계 문서가 요구하는 `판정 순서 전체` 보정 범위가 충분히 드러나지 않는다.

설계자 보완 요청:
- 엔진 보정 작업은 아래 전체 시퀀스를 맞추는 것으로 정의해야 한다.
1. 착지
2. 고정
3. 라인 수 계산
4. Perfect Clear 판정
5. Back-to-Back 판정
6. Combo 판정
7. 점수 계산
8. HUD 갱신
9. 다음 블록 스폰
10. 스폰 실패 시 게임 오버
- 즉, 점수 항목 일부 보정이 아니라 `판정 순서 전체 정합화`로 작업 범위를 다시 적는 것이 맞다.

### 4.3 Marathon 동점 규칙 후속 조치 범위
판단:
- `수용`, 단 영향 범위 확대 명시 필요

회신:
- Marathon 동점 규칙을 D4 전에 계약으로 잠가야 한다는 판단은 타당하다.
- 그러나 현재 후속 조치는 `BE/DB 계약 고정`으로만 적혀 있어 범위가 좁다.
- 설계 기준상 이 항목은 API 정렬, DB 조회, 테스트 데이터뿐 아니라 QA 기대결과까지 함께 고정돼야 한다.

설계자 보완 요청:
- 후속 조치는 `BE/DB/QA` 또는 `계약/조회/테스트 기대결과` 수준으로 확장 표기하는 편이 정확하다.
- 이 규칙은 구현 완료 뒤 맞추는 항목이 아니라, 랭킹 개발 착수 전 테스트 기준까지 함께 잠가야 한다.

## 5. 설계자 최종 의견
- 개발자의 `D1 재오픈 불필요`, `D2 전 선행 보정 필요` 판단은 수용한다.
- 현재 우선순위를 `엔진 규칙 보정 -> 분석 이벤트 -> 모바일 세로 -> 접근성 최소 옵션`으로 둔 것도 타당하다.
- 다만 최종 전달 문서로 사용할 때는 아래 3가지를 보완한 버전이 더 적절하다.
  - 분석 이벤트 미충족 근거를 `단일 파일 부재`가 아니라 `앱 코드 전체 미연결`로 확대
  - 엔진 보정 범위를 `점수 체계 일부`가 아니라 `판정 순서 전체 정합화`로 명시
  - Marathon 동점 규칙 영향 범위를 `BE/DB`에서 `BE/DB/QA`까지 확대

## 6. 한 줄 회신
- 개발자 검토 방향은 수용하되, 이벤트 근거 범위, 엔진 보정 범위, 랭킹 규칙 영향 범위를 더 엄밀하게 적어 전달하는 것이 설계 기준상 맞다.
