# 설계대비 개발진행현황 리포트

## 1. 목적
- 본 문서는 현재 저장소 구현 상태를 `2.설계문서`와 대조해 설계 기준 대비 개발 진행 현황을 보고하기 위한 리포트다.
- 기준일은 `2026-03-11`이다.

## 2. 종합 판단
### 2.1 전체 상태
- 현재 개발 단계는 `D0 완료, D1 착수 단계`로 판단한다.
- 설계 문서 기준 `개발 기반 확정`은 충족했다.
- 그러나 사용자 관점의 핵심 MVP 기능은 아직 대부분 미착수 또는 부분구현 상태다.

### 2.2 한 줄 평가
- `기반은 준비되었고, 실제 서비스 기능은 아직 시작 단계다.`

## 3. 핵심 결론
### 완료된 영역
- 모노레포 구조, 앱 분리, 패키지 분리, DB 마이그레이션/시드 명령, 기본 테스트 명령이 준비되었다.
- 공개 웹 랜딩과 운영 프리뷰의 최소 화면/선택자가 준비되었다.
- API는 `health`, `bootstrap`, `game-sessions 생성`, `settings`, `test gate` 골격이 있다.
- game-engine은 `7-bag`, 보드 생성, 이동/회전/충돌/라인 제거 기초 함수와 단위 테스트가 있다.

### 부분구현 영역
- F-01 첫 진입 및 빠른 시작
- F-02 코어 게임플레이
- F-07 데일리 챌린지 및 미션
- F-09 설정 및 접근성
- F-11 운영 백오피스

### 미착수 또는 실질 미구현 영역
- F-03 모드 운영
- F-04 결과 화면 및 재도전
- F-05 개인 기록 및 닉네임 등록
- F-06 랭킹
- F-08 업적 및 장기 목표
- F-10 공유
- F-12 부정행위 및 운영 보호

## 8. 주요 리스크
### R-01
- 설계서상 Must 기능은 많지만 현재 런타임 구현은 랜딩/API 스캐폴드 위주라, 외부 이해관계자가 진행률을 과대평가할 위험이 있다.

### R-02
- DB 설계가 앞서 있고 서비스 레이어 구현이 뒤처져 있어, 후속 개발에서 DTO/엔터티/실제 응답 간 불일치가 생길 수 있다.

### R-03
- `settings`는 현재 메모리 저장이라 설계의 `다음 세션에도 유지` 완료 조건을 충족하지 못한다.

### R-04
- 결과 제출 API와 검증 상태 전이 계약이 아직 코드 타입으로 고정되지 않아 D3에서 프론트엔드, 백엔드, QA 해석 차이가 날 가능성이 높다.

### R-05
- 운영 프리뷰가 존재해 F-11이 진행된 것처럼 보일 수 있으나, 실제 관리 기능은 아직 없다.

## 9. 설계자 권고
### 즉시 우선순위
1. D1 범위의 게임 화면, 입력 루프, 결과 화면을 먼저 완성한다.
2. D3 범위의 `finish` API 계약과 멱등성 규칙을 `shared-types`에 먼저 고정한다.
3. settings를 DB 영속 저장으로 전환해 F-09의 최소 기준을 맞춘다.
4. 모드/랭킹/닉네임/데일리 API는 스키마보다 서비스 계약부터 잠근 뒤 구현한다.

### 관리 권고
1. 이후 진행 보고에서는 `스키마 존재`와 `사용자 기능 완료`를 반드시 분리해 표기한다.
2. 단계 보고는 `D0 완료`, `D1 진행 중`처럼 단계 중심으로 하되, F-기능 완성률은 별도 표로 관리한다.

## 10. 최종 결론
- 현재 저장소는 설계 문서 기준으로 `개발을 시작할 수 있는 기반`은 갖췄다.
- 그러나 `서비스 기능이 구현되었다`고 보고할 단계는 아니다.
- 가장 정확한 현재 표현은 `D0 완료, D1 코어 플레이 구현 진행 중`이다.
