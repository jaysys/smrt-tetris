# 개발관리 개발계획서

## 1. 문서 목적
본 문서는 `2.설계문서`를 기준으로 실제 개발 착수 이후 필요한 개발 관리 기준을 정의한다.

이 문서의 목적은 다음과 같다.
- 개발 범위와 단계별 목표를 관리 관점에서 통제한다.
- 프런트엔드, 백엔드, QA가 같은 완료 기준으로 움직이도록 한다.
- 제3의 테스터가 독립적으로 검증 가능한 상태를 각 단계의 기본 완료 조건으로 둔다.
- 설계 변경, 일정 변경, 품질 미달 항목이 생겼을 때 의사결정 기준을 남긴다.

## 2. 개발 추진 원칙
- 설계 문서에 없는 내용은 임의 구현하지 않고 이슈로 등록 후 결정한다.
- 코어 게임 로직은 UI와 분리된 독립 모듈로 구현한다.
- 모든 기능은 테스트 가능한 단위로 나누어 구현한다.
- 기능 구현과 동시에 자동 검증 가능성(testability)을 산출물로 함께 제공한다.
- 각 단계 완료 시점마다 제3자 테스트 가능 상태를 확보해야 한다.
- 구현 완료와 테스트 완료를 같은 의미로 보지 않는다.
- 운영 문서와 실제 코드가 불일치하면 코드가 아니라 문서를 먼저 점검한다.
- 개발 진행 중 발생한 결정, 변경, 이슈, 완료 상태는 관련 관리 문서에 즉시 반영해 항상 현행 상태를 유지한다.
- 문서 현행화는 선택 작업이 아니라 각 작업 완료 조건의 일부로 간주한다.
- 문서의 `갱신자` 표기가 필요한 경우 표준 표기는 항상 `Jaeho`로 통일한다.

## 3. 개발 범위 기준

- 게스트 즉시 플레이
- Marathon, Sprint, Daily Challenge
- 게임 세션 생성과 결과 제출
- 닉네임, 개인 최고 기록, 랭킹
- 공유 링크
- 미션, 업적, 보상
- 운영 백오피스 기본 기능
- 접근성, 모바일 대응, 분석 이벤트, 기본 부정행위 방지

### 제외 범위
- 실시간 대전
- 친구 비교
- 상점, 결제, 시즌 패스
- PWA 설치 유도
- 오프라인 모드

## 6. 작업 분류 체계
개발 작업은 아래 단위로 관리한다.

| 분류 | 설명 |
|---|---|
| `FE` | 프런트엔드 화면 및 UX |
| `GE` | 게임 엔진 |
| `BE` | API 및 서버 로직 |
| `DB` | 스키마, 마이그레이션, 시드 |
| `ADMIN` | 운영 백오피스 |
| `QA` | 테스트 케이스 실행 및 결함 검증 |
| `DOC` | 문서 보완 및 계약 정리 |
| `OPS` | 배포, 로그, 환경 구성 |

작업 ID 예시:
- `GE-001 7-bag 생성기 구현`
- `BE-003 결과 제출 멱등 처리`
- `QA-010 Sprint 랭킹 정렬 검증`

## 7. 작업 상태 관리
모든 작업은 아래 상태로 관리한다.

| 상태 | 의미 |
|---|---|
| `TODO` | 착수 전 |
| `IN_PROGRESS` | 구현 중 |
| `BLOCKED` | 외부 결정 또는 선행 작업 대기 |
| `DEV_DONE` | 구현 완료, 자체 확인 완료 |
| `QA_READY` | 테스트 가능 상태 |
| `QA_FAIL` | 테스트 실패 |
| `DONE` | 테스트 통과 및 문서 반영 완료 |

상태 전이 원칙:
- `DEV_DONE`으로 올리기 전에 최소 단위 테스트 또는 수동 검증 결과가 있어야 한다.
- `DONE`은 QA 확인과 문서 반영까지 끝난 상태만 허용한다.

## 8. 품질 게이트
### 구현 완료 기준
- 설계 문서 기준 동작이 재현된다.
- 예외 처리와 오류 응답이 설계와 일치한다.
- 필요한 로그 또는 디버깅 포인트가 있다.
- 자동화 가능한 선택자, 상태 노출, 테스트 제어 수단이 함께 제공된다.

### QA 착수 기준
- 테스트 대상 기능이 배포 또는 로컬 실행 가능 상태다.
- 테스트 데이터가 준비되어 있다.
- 기대 결과가 문서상 판정 가능하다.
- 수동 테스트와 자동 테스트가 같은 기준 데이터와 같은 시간 제어 방식을 사용한다.

### 단계 종료 기준
- 해당 단계 Must 기능 구현 완료
- 해당 단계 핵심 테스트 통과
- blocker 결함 0건
- 문서와 구현 간 차이점 기록 완료
- 단계별 자동화 요구사항 반영 상태 기록 완료

## 8.1 자동화 대응 기준
### 자동화 도구 기준
- 웹 E2E 기본 도구는 `Playwright`를 기준으로 한다.
- 게임 엔진 단위 테스트는 `Vitest`를 기준으로 확대한다.
- API 통합 테스트는 API 런타임과 분리 가능한 테스트 구성을 별도로 둔다.

### 테스트 가능성 구현 기준
- 핵심 CTA와 상태 표시 요소에는 안정적인 `data-testid`를 부여한다.
- 시간 의존 기능은 `TEST_CLOCK_ISO_UTC` 기준으로 제어 가능해야 한다.
- 테스트 전용 기능은 `TEST_MODE` 또는 동등한 환경 게이트 하에서만 노출한다.
- 결과 제출, 보상 수령, 운영 설정 반영은 반복 실행 가능한 초기화 절차를 제공한다.
- 랜덤 요소가 있는 기능은 테스트 환경에서 시드 고정 또는 입력 재생이 가능해야 한다.

### 테스트 전용 인터페이스 기준
- 테스트 DB 초기화
- 시드 데이터 재적재
- 테스트 시계 주입
- 테스트용 관리자 토큰 또는 로그인 수단
- 비정상 결과 재현용 입력 세트 또는 샘플 데이터

### 자동화 산출물 기준
- 실패 시 스크린샷, 응답 본문, 콘솔 로그, 네트워크 로그를 저장할 수 있어야 한다.
- 자동화 실패 산출물 저장 경로와 수집 방식은 테스트 구현 시 문서화한다.
- 자동화 명령은 최소 `pnpm test`, `pnpm test:e2e` 단일 진입점을 제공한다.

## 9. 이슈 및 변경 관리
### 설계 이슈
- 설계 문서 해석이 모호하면 구현 전에 `설계 이슈`로 등록한다.
- 설계 변경이 확정되면 기능설계서, 시스템설계서, API명세서, 테스트계획서를 함께 점검한다.
- 설계 변경이 코드에 반영되면 같은 날 관련 문서도 현행화한다.

### 개발 이슈
- 버그, 성능 문제, 일정 지연은 `개발 이슈`로 분류한다.
- blocker 이슈는 같은 날 공유하고 우선순위를 재조정한다.
- 해결된 이슈는 해결 시점, 원인, 영향 범위, 후속 조치를 이슈 문서에 즉시 기록한다.

### 범위 변경
- MVP 범위를 벗어나는 요청은 즉시 `후속 범위`로 분리한다.
- 현재 단계 일정과 충돌하면 기본적으로 차기 단계로 이월한다.
- 범위 변경이 발생하면 작업관리대장, 의사결정기록서, 관련 설계 문서를 함께 현행화한다.

## 9.1 문서 현행화 관리 원칙
- 작업 상태가 바뀌면 같은 날 `작업관리대장`을 갱신한다.
- blocker, 결함, 재현 불가 이슈가 생기면 즉시 `이슈/결함대장`에 등록한다.
- 설계 해석 변경, 기술 선택 변경, 범위 조정이 발생하면 `의사결정기록서`를 갱신한다.
- 테스트 결과가 나오면 Pass/Fail 여부와 근거를 관리 문서에 기록한다.
- 문서가 최신이 아니면 해당 작업은 `DONE`으로 간주하지 않는다.
- 사용자에게 완료 보고, 완료 확인 요청, 다음 단계 진행 여부 확인을 하기 전에는 관련 관리 문서를 먼저 현행화한다.
- 진행 보고의 기준 문서는 `작업관리대장`으로 하며, 단계 상태는 `개발관리 개발계획서`와 일치해야 한다.
