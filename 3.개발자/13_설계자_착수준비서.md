# 설계자 착수준비서

## 1. 목적
- 본 문서는 `2.설계문서`를 검토한 뒤, 현재 저장소 기준으로 설계자가 즉시 준비해야 할 업무를 정리한 실행 문서다.
- 목표는 `설계 수정`이 아니라 `개발 착수 충돌 방지`, `구현 계약 고정`, `인수 기준 명확화`다.

## 2. 현재 판단
### 2.1 설계 문서 상태
- `2.설계문서`는 기능, 시스템, API, DB, 테스트, QA 가이드까지 포함하고 있어 개발 착수 가능 상태다.
- `3.개발자/06_설계자_최종확인.md` 판단대로 설계 본문 자체의 추가 수정은 현재 필수는 아니다.

### 2.2 저장소 구현 상태
- 현재 저장소는 `D0 개발 기반 확정` 수준이다.
- 구현 반영 범위는 공개 웹/운영 프리뷰 스캐폴드, 부트스트랩 API, 세션 생성 API, 설정 API, 초기 DB 스키마, game-engine 기초 유틸까지다.
- 미구현 핵심 범위는 코어 플레이 UI, 결과 화면/재도전, 결과 제출 멱등 처리, 랭킹, 공유, 운영 기능 대부분이다.

### 2.3 설계자 관점 결론
- 지금 설계자가 해야 할 일은 새로운 기능을 더 설계하는 것이 아니다.
- 아래 5개 축의 계약을 개발 전에 명확히 잠그는 일이다.
  - D1 코어 플레이 판정 기준
  - D3 결과 제출/검증 계약
  - 테스트 제어 인터페이스
  - 문서 간 상태 불일치 정리
  - 단계별 인수 기준 고정

## 3. 설계자가 즉시 준비할 업무
### 3.1 D1 게임 엔진 인수 기준 패키지 고정
- 기준 문서:
  - `2.설계문서/01_기능설계서.md`
  - `2.설계문서/12_게임규칙_수치테이블.md`
- 준비 업무:
  - 블록 스폰 좌표, 회전 상태, 벽차기, DAS/ARR, 락 딜레이, 점수 규칙을 코드 상수와 1:1로 매핑한다.
  - `GE-002` 완료 기준을 `수동 플레이 가능`이 아니라 `정량 규칙 테스트 통과`로 구체화한다.
  - D1 검증 항목을 최소 단위 테스트 목록으로 내려서 개발자와 QA가 같은 기준을 쓰게 한다.
- 산출물:
  - 엔진 검증 체크리스트
  - 엔진 단위 테스트 우선순위 목록

### 3.2 D3 결과 제출/검증 계약 고정
- 기준 문서:
  - `2.설계문서/02_시스템설계서.md`
  - `2.설계문서/03_API명세서.md`
  - `2.설계문서/04_DB설계서.md`
- 준비 업무:
  - `POST /v1/game-sessions/{sessionId}/finish` 요청/응답 DTO를 `shared-types`에 선반영한다.
  - `ACCEPTED | FLAGGED | REJECTED` 전이 규칙과 사유 코드를 코드 enum 수준으로 고정한다.
  - `Idempotency-Key`, `payload hash`, `sessionId당 1회 공식 반영` 규칙을 API/DB/테스트에서 동일하게 해석하도록 예시를 정리한다.
  - Sprint 최소 가능 시간, 입력 빈도, 체크포인트 누락 기준을 서버 검증 규칙 표로 별도 추출한다.
- 산출물:
  - 결과 제출 계약서 초안
  - 검증 상태 전이표
  - 멱등성 시나리오 표

### 3.3 테스트 제어 인터페이스 확정
- 기준 문서:
  - `2.설계문서/14_QA실행가이드.md`
  - `3.개발자/07_개발관리_개발계획서.md`
- 준비 업무:
  - `TEST_CLOCK_ISO_UTC`를 모든 시간 의존 기능의 단일 표준으로 확정하고 API/스케줄러/랭킹 계산에 동일 적용되게 한다.
  - `TEST_MODE` 하에서 필요한 최소 테스트 인터페이스 목록을 확정한다.
  - 테스트 DB 초기화, 시드 재적재, 관리자 접근, 비정상 기록 재현 방식의 표준 절차를 고정한다.
- 산출물:
  - 테스트 훅 목록
  - 자동화 전용 환경 계약

### 3.4 문서 간 상태 불일치 정리
- 현재 확인된 불일치:
  - `3.개발자/08_작업관리대장.md`는 `DB-001`을 `db:seed 검증 완료`로 기록한다.
  - `4.테스터/05_현재_테스트가능범위_식별서.md`는 `db:seed 실패`를 기록한다.
- 준비 업무:
  - 실제 실행 결과를 재확인하고 하나의 기준으로 문서를 통일한다.
  - `QA_READY` 기준과 실제 실행 기준이 어긋나는 항목을 정리한다.
- 산출물:
  - 상태 정정 메모 또는 관리 문서 수정

### 3.5 단계별 인수 기준 명문화
- 기준 문서:
  - `2.설계문서/06_MVP개발계획서.md`
  - `3.개발자/07_개발관리_개발계획서.md`
  - `4.테스터/05_현재_테스트가능범위_식별서.md`
- 준비 업무:
  - D1, D2, D3 단계마다 설계자가 승인해야 하는 필수 인수 항목을 3~5개 수준으로 고정한다.
  - 구현 완료, QA 가능, 최종 완료를 혼동하지 않도록 상태 전이 기준을 더 짧고 명확하게 정리한다.
- 산출물:
  - 단계별 설계 인수 체크리스트

## 4. 역할별 전달 준비
### 프론트엔드에 전달할 내용
- 랜딩 이후 게임 화면, 결과 화면, 모드 선택 화면의 필수 `data-testid`
- 모바일 세로 HUD 우선순위
- 결과 화면 CTA 4종 고정 순서와 최초 가시 영역 규칙

### 백엔드에 전달할 내용
- 부트스트랩, 세션 생성, 결과 제출, 랭킹, 닉네임, Daily Challenge API 계약
- 시간 기준 UTC 저장/KST 집계 원칙
- 검증 상태와 멱등성 예외 처리 규칙

### QA에 전달할 내용
- D1부터의 최소 실행 가능 시나리오
- `TEST_CLOCK_ISO_UTC` 사용 규칙
- 단계별 테스트 착수 조건과 보류 조건

### 운영/관리 화면 담당에 전달할 내용
- D5 이전에는 프리뷰 스캐폴드 수준임을 명시
- 챌린지 예약, 공지, 플래그 조치의 최소 관리 필드 정의

## 5. 설계자 우선순위 업무 순서
1. 결과 제출 계약과 검증 상태 전이표 고정
2. D1 엔진 인수 체크리스트 고정
3. 테스트 훅과 시간 제어 표준 고정
4. 문서 간 상태 불일치 정리
5. 단계별 인수 체크리스트 배포

## 6. 현재 시점 리스크
- 설계 문서는 충분하지만 코드 계약 타입이 아직 부분 반영 상태다.
- D1/D3 경계에서 문서 해석 차이로 프론트엔드, 백엔드, QA가 서로 다른 완료 판단을 할 가능성이 있다.
- 관리 문서와 테스터 문서의 상태 불일치가 계속되면 `QA_READY` 의미가 약해진다.

## 7. 최종 판단
- 설계 문서 재작성은 지금 우선순위가 아니다.
- 설계자의 실제 다음 업무는 `구현 전 계약 잠금`, `검증 기준 배포`, `문서 상태 정합성 회복`이다.
- 위 3가지를 끝내면 개발팀은 D1~D3을 설계 이슈 없이 밀어붙일 수 있다.
