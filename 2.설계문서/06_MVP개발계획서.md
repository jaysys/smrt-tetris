# MVP개발계획서

## 1. 개발 원칙
- 플레이 경험을 먼저 완성하고 부가 요소를 붙인다.
- 코어 게임 로직은 UI와 분리된 독립 모듈로 먼저 개발한다.
- 각 단계는 테스터가 바로 확인 가능한 완료 기준을 가져야 한다.
- MVP에 포함되지 않는 기능은 설계만 남기고 구현은 미룬다.

## 2. 개발 순서 개요
| 단계 | 목표 | 핵심 산출물 |
|---|---|---|
| Phase 0 | 정책/브랜딩/기술 기반 확정 | 프로젝트 기본 구조, 설계 확정 |
| Phase 1 | 코어 플레이 가능 상태 | 엔진, 입력, 점수, 결과 화면, 모바일 기본 HUD, 핵심 퍼널 이벤트 |
| Phase 2 | MVP 모드 완성 | Marathon, Sprint, Daily Challenge, 모바일 세로 검증 |
| Phase 3 | 기록과 경쟁 완성 | 닉네임, 개인 최고 기록, 랭킹, 접근성 최소 옵션 |
| Phase 4 | 리텐션과 운영 완성 | 미션, 업적, 운영 백오피스, 난이도 운영 규칙 |
| Phase 5 | 품질 고도화 및 출시 준비 | 분석 고도화, 보안, 반응형/접근성 최종 보강 |

## 3. 단계별 상세 계획
## 3.1 Phase 0: 기반 확정
### 개발 항목
- 독자 IP 전제 브랜딩 placeholder 적용
- 모노레포 초기화
- 공통 타입/환경변수 구조 정의
- DB 마이그레이션 기본 틀 생성

### 완료 조건
- 개발 저장소 구조가 생성되어 팀원별 병렬 개발이 가능하다.
- 설계 문서 기준 API, DB, 모듈 명칭이 코드 구조에 반영된다.

## 3.2 Phase 1: 코어 플레이 가능 상태
### 개발 항목
- 게임 보드 렌더러
- 7-bag 블록 생성
- 이동/회전/드롭/충돌 판정
- Ghost Piece 착지 가이드 표시
- 라인 제거 및 점수 계산
- 게임 오버 판정
- 결과 화면 및 재도전
- 모바일 세로 기본 HUD/조작부 레이아웃
- 핵심 퍼널 이벤트 수집
  - `landing_view`
  - `quick_start_click`
  - `game_start`
  - `game_finish`
  - `retry_click`

### 선행 조건
- Phase 0 완료

### 완료 조건
- 브라우저에서 Marathon 기본 플레이가 가능하다.
- 키보드와 모바일 터치 조작이 최소 동작한다.
- Ghost Piece가 기본값 `ON`으로 표시되고, 현재 블록과 구분되는 최대한 연한 착지 가이드로 보인다.
- 결과 화면에서 재도전이 가능하다.
- 모바일 390 x 844, 360 x 800 기준에서 HUD와 조작부가 겹치지 않는다.
- 코어 퍼널 이벤트가 로그 또는 분석 적재 기준으로 확인 가능하다.

## 3.3 Phase 2: MVP 모드 완성
### 개발 항목
- Marathon 레벨 증가 규칙
- Sprint 40라인 타임어택
- Daily Challenge 1종
- 모드 선택 화면
- 개인 최고 기록 로컬/서버 저장
- 모바일 세로 레이아웃과 모드별 화면 검증

### 완료 조건
- 모드 3종의 종료 조건과 결과 기준이 분리되어 동작한다.
- Daily Challenge가 설정값 기반으로 동작한다.
- 모바일 세로 화면에서 모드별 HUD와 조작부 레이아웃이 유지된다.

## 3.4 Phase 3: 기록과 경쟁 완성
### 개발 항목
- 게스트 토큰 발급
- 게임 세션 생성/결과 제출 API
- 닉네임 등록
- 모드별 일간/주간/전체 랭킹
- 공유 링크 생성
- 접근성 최소 옵션
  - 고대비 모드
  - 시각 효과 축소
  - HUD 가독성 유지

### 완료 조건
- 공식 기록이 서버 검증 후 랭킹에 반영된다.
- 닉네임 등록 후 랭킹 조회가 가능하다.
- 접근성 최소 옵션이 게임 화면과 결과 화면에 반영된다.

## 3.5 Phase 4: 리텐션과 운영 완성
### 개발 항목
- Daily 보상 수령
- 반복 미션 3종
- 업적 5종
- 운영자 Daily Challenge 설정 화면
- 공지사항 관리
- 부정행위 플래그 검토 화면
- Daily Challenge 난이도 운영 가이드

### 완료 조건
- 운영자가 다음 날 챌린지를 설정할 수 있다.
- 플레이 결과가 미션과 업적에 반영된다.
- 운영자가 일별 챌린지 난이도를 과도하게 편향시키지 않도록 운영 기준을 참조할 수 있다.

## 3.6 Phase 5: 품질 고도화 및 출시 준비
### 개발 항목
- 분석 이벤트 고도화
- 레이트 리밋, 결과 검증 강화
- 공지/점검 페이지
- 반응형/접근성 최종 보강

### 완료 조건
- 주요 브라우저 QA를 통과한다.
- 분석 대시보드에서 핵심 KPI 측정이 가능하다.

## 4. 병렬 개발 권장 구조
| 담당 영역 | 선행 단계 | 병렬 가능 시점 |
|---|---|---|
| 게임 엔진 | Phase 0 | 즉시 |
| UI/랜딩/결과 화면 | Phase 0 | 즉시 |
| API/세션/결과 검증 | Phase 1 일부 완료 후 | Phase 1 중반 |
| 랭킹/프로필 | Phase 2 | Phase 2 후반 |
| 백오피스 | Phase 3 | Phase 3 후반 |
| 분석/운영 | Phase 4 | Phase 4 후반 |

## 5. MVP 범위에서 제외
- 친구 비교
- 시즌 패스
- 실시간 대전
- Zen 모드
- 상점 및 결제
- PWA 설치 유도

## 6. 개발 우선순위 판단 기준
- 플레이를 막는 기능은 최우선
- 기록과 랭킹 루프는 두 번째
- 재방문 동기와 운영 도구는 세 번째
- 수익화와 확장 기능은 후순위
