## 1. 문서 목적
- 본 문서는 웹 화면 직접 수행형 테스트를 백그라운드 자동 테스트로 전환할 수 있도록 개발팀에 요청하는 구현 요구사항을 정리한다.
- 테스터 관점의 핵심 목표는 `사람이 브라우저를 직접 열어 반복 확인하는 케이스를 CI 또는 로컬 헤드리스 실행으로 재현 가능하게 만드는 것`이다.

## 2. 배경
- 현재 저장소는 D0 스캐폴드 중심 상태이며, 웹/운영 화면과 API는 실제 기능 구현 전 단계가 많다.
- 현재 기준으로는 수동 테스트 문서 작성은 가능하지만, 웹 화면 직접 수행형 케이스를 자동으로 재현할 실행 기반이 부족하다.
- 따라서 개발팀은 기능 구현과 동시에 `테스트 가능성(testability)`을 산출물로 함께 제공해야 한다.

### 4.2 상태 노출 규칙
- 화면에서만 보이는 상태가 아니라 자동화가 읽을 수 있는 상태값을 함께 노출해야 한다.
- 예:
  - 결과 검증 상태 `ACCEPTED`, `FLAGGED`, `REJECTED`
  - 재전송 대기 상태
  - 보상 수령 가능/완료 상태
  - 랭킹 반영 여부
  - 화면 구조 상태 `sheet`, `modal`, `page`
  - 화면 대비 상태 `default`, `high`

### 4.2A UI 구조 자동 판정 규칙
- 자동화 코드는 텍스트 내용만이 아니라 구조를 함께 판정할 수 있어야 한다.
- 최소 확인 항목:
  - 랜딩 first-fold에서 `start-button`, `mode-button`, `ranking-button` 동시 노출
  - `landing-hero`가 `daily-banner`보다 시각상 우선 영역에 존재
  - `control-dock`이 viewport 하단 safe area 위에서 고정 노출
  - `result-cta-grid`가 first-fold 내에서 4개 CTA를 모두 포함
  - `settings-sheet`, `nickname-sheet`가 모바일에서 전체 페이지 전환 없이 열림

### 4.3 게임 화면 자동화 지원
- 게임 시작, 일시정지, 재개, 종료를 테스트 코드에서 직접 호출하거나 재현할 수 있는 인터페이스가 필요하다.
- 최소 1개 방식은 반드시 제공해야 하며, MVP 우선순위는 아래와 같이 요청한다.
- 1순위: 시드 고정 + 입력 재생
- 2순위: 테스트 전용 디버그 훅
- 3순위: 테스트 전용 상태 주입 API
- 우선순위 선정 이유:
  - 엔진 재현성과 회귀 안정성을 가장 높게 확보할 수 있다.
  - 운영 환경과 테스트 환경의 분리를 유지하기 쉽다.
  - 헤드리스 E2E와 단위/통합 테스트 간 재사용성이 높다.
- 추가 UI 테스트성 요구:
  - 게임 화면은 safe area 계산 후 실제 배치된 `hold-slot`, `next-queue`, `control-dock` 경계를 자동화에서 읽을 수 있어야 한다.
  - 조작 버튼은 크기 판정이 가능하도록 개별 버튼 선택자를 유지해야 한다.

## 5. API 및 백엔드 자동화 대응 요구사항
### 5.1 테스트 제어 기능
- 아래 제어 수단은 테스트 환경에서만 활성화되어야 한다.
- 테스트 DB 초기화 명령
- 시드 데이터 재적재 명령
- 테스트 시계 주입
- 운영자 테스트 로그인 또는 토큰 발급
- 비정상 결과 샘플 주입 또는 재현 가능한 검증 입력 세트

표준 경로 요구:
- 테스트 제어 기능은 `TEST_MODE=true` 환경에서만 활성화되어야 한다.
- 명령형 표준:
  - `pnpm db:reset`
  - `pnpm db:migrate`
  - `pnpm db:seed`
  - `pnpm test:e2e`
- API형 표준:
  - `/v1/test/*` 또는 동등한 테스트 전용 엔드포인트
- 운영 환경에서는 위 API가 비활성화되거나 접근 차단되어야 한다.

### 5.2 응답 검증 요구사항
- 모든 핵심 API는 문서화된 응답 구조를 유지해야 한다.
- 자동화 검증 필드는 엔드포인트별로 아래처럼 구분한다.

엔드포인트 공통 검증 필드:
- `success`
- 실패 시 `error.code`
- 필요 시 `meta.retryable`

결과 제출 API `POST /game-sessions/{id}/finish` 필수 검증 필드:
- `data.validationStatus`
- `data.personalBest`
- `data.leaderboardUpdated`
- `meta.idempotentReplay`
- `meta.retryable`

랭킹 API `GET /leaderboards` 필수 검증 필드:
- `data.mode`
- `data.period`
- `data.entries`
- `data.myRank`

Daily Challenge API `GET /daily-challenge`, `POST /daily-challenge/{id}/claim` 필수 검증 필드:
- `data.challengeId`
- `data.claimed`
- `data.myProgress`
- 실패 시 `error.code`

닉네임 API `POST /profiles/nickname` 필수 검증 필드:
- `data.profileId`
- `data.nickname`
- `data.status`
- 실패 시 `error.code`

운영 API `/admin/*` 필수 검증 필드:
- 성공 시 주요 식별자와 상태 필드
- 실패 시 `error.code`
- 권한 오류 시 `401` 또는 `403` 구분

### 5.3 실패 산출물 저장 규약
- 실패 산출물은 로컬과 CI에서 동일한 경로 규칙으로 저장되어야 한다.

저장 경로 규약:
```text
artifacts/
  e2e/
    <run-date>/
      <suite-name>/
        <test-case-id>/
          screenshot.png
          response.json
          console.log
          network.har
```

파일명 규약:
- 스크린샷: `screenshot.png`
- 응답 본문: `response.json`
- 콘솔 로그: `console.log`
- 네트워크 로그: `network.har`
- 추가 첨부가 있으면 `extra-01.*`, `extra-02.*` 형식 사용

보존 범위:
- 로컬: 최근 10회 실행 또는 7일
- CI: 최근 성공 5회, 실패 실행 20회 또는 30일
- 최소 보존 대상은 `P0` 실패 케이스 전부다

### 5.4 테스트용 명령 요구사항
- 개발팀은 아래 명령을 실제 동작 가능한 상태로 제공해야 한다.

```bash
pnpm install
pnpm db:reset
pnpm db:migrate
pnpm db:seed
pnpm test
pnpm test:e2e
```

- `pnpm test:e2e`는 최소한 P0 웹 시나리오를 백그라운드로 실행해야 한다.

## 6. 우선 자동화 대상
### 6.1 P0 자동화 필수 대상
- 랜딩 진입 후 즉시 플레이
- 핵심 조작 후 한 판 종료

- 결과 제출 성공과 개인 최고 기록 반영

- 동일 키 다른 payload 충돌 차단

- KST 자정 경계 Daily 전환
- 주간 랭킹 월요일 경계 전환
- 랭킹 API 실패 안내
- 설정 저장 실패 롤백
- 운영 권한 오류
- 모바일 레이아웃 회귀

## 8. 개발팀 구현 원칙 요청
- 기능 구현 시 테스트 코드를 나중에 붙이는 방식이 아니라, 화면/상태/API를 자동 검증 가능한 형태로 같이 설계해야 한다.
- 임시 텍스트 탐색 기반 자동화는 금지하고, 반드시 고정 선택자를 제공해야 한다.
- 랜덤 요소가 있는 기능은 테스트 환경에서 결정적 실행이 가능해야 한다.
- 테스트 전용 기능은 운영 환경에서 비활성화되어야 한다.
- 문서에 명시된 명령, API, 시간 제어 방식이 실제 코드와 다르면 개발 완료로 보지 않는다.
- 사용자 화면에는 기술 검증 카드, 내부 상태 설명, 임시 영문 문구가 남지 않도록 금칙 UI를 자동 회귀 대상으로 관리해야 한다.

## 9. 인수 기준
- 아래 조건을 만족하면 테스터 요구사항이 수용된 것으로 본다.
- `pnpm install`은 `registry.npmjs.org` 또는 사내 패키지 미러에 접근 가능한 네트워크 환경을 전제로 성공해야 한다.
- 위 전제가 충족된 환경에서 `pnpm install` 이후 로컬에서 앱과 테스트 명령이 실제 실행된다.
- `pnpm test:e2e`로 P0 웹 시나리오가 헤드리스로 실행된다.
- 실패 시 스크린샷, 로그, 응답 캡처가 자동 저장된다.
- `TEST_CLOCK_ISO_UTC`로 Daily/주간 경계 테스트가 재현된다.
- 결과 제출 멱등성과 부정행위 차단 케이스가 자동 검증된다.
- 운영자 시나리오 최소 1건 이상이 자동화된다.
- 랜딩 first-fold, 게임 화면 safe area, 결과 화면 CTA 4개 first-fold, 설정/닉네임 바텀시트 구조가 자동 판정된다.

## 10. 개발팀 확인 요청 사항
- E2E 도구는 Playwright를 기준으로 제안한다.
- API 통합 테스트는 Nest 환경에서 별도 구성한다.
- 게임 엔진은 Vitest 단위 테스트를 확대한다.
- CI에서는 최소 `lint -> unit test -> e2e P0` 순서로 수행한다.
- 게임 자동화 지원 방식 우선순위는 `시드 고정 + 입력 재생 -> 디버그 훅 -> 상태 주입 API` 순서로 동의한다.

## 11. 결론
- 테스터 입장에서는 `기능 구현 완료`보다 `자동 검증 가능한 구현 완료`가 더 중요하다.
- 개발팀이 위 요구사항을 수용하지 않으면 웹 화면 직접 수행형 케이스는 계속 수동 검증에 의존하게 되고, 회귀 비용과 릴리스 리스크가 커진다.
- 따라서 본 문서의 요구사항은 선택 보강이 아니라 MVP 품질 확보를 위한 필수 개발 대응 항목으로 전달한다.
