# UI화면설계서

## 1. 목적
- 본 문서는 사용자 화면의 UI 구조, 정보 위계, 모바일 세로 배치, 상호작용 규칙을 개발자가 임의 해석 없이 구현할 수 있도록 고정하는 전용 화면설계서다.
- 현재 구현 화면에 포함된 기술 검증용 구성, 임시 텍스트, 개발 편의 중심 배치를 사용자용 제품 화면으로 교정하는 것이 목적이다.

### 3.2 적용 해석
- Tetris 공식 제품 계열에서는 `빠른 한 판 시작`, `Marathon`, `Quick Play`, `Daily Challenges`, `직관적 터치/온스크린 컨트롤`, `개인화`가 핵심 축으로 반복 확인된다.
- Apple 가이드에서는 `스크린 안에 주 콘텐츠가 들어와야 함`, `44pt 이상 터치 타깃`, `컨트롤은 수정 대상 콘텐츠 가까이에 배치`, `게임 UI는 safe area 안에서 읽히고 눌려야 함`이 핵심이다.
- Android 가이드에서는 `작은 텍스트 4.5:1 대비`, `큰 텍스트 3:1 대비`, `최소 48dp 터치 타깃`, `장식 요소를 제외한 UI 요소 설명`이 핵심이다.

### 3.3 채택 원칙
- 본 서비스는 외부 레퍼런스의 `정보 구조`와 `조작 배치 원칙`을 참조한다.
- 라이선스 미확보 상태이므로 공식 Tetris의 브랜드 자산, 로고, 전용 시각 표현, 트레이드 드레스는 모사하지 않는다.
- 따라서 본 문서는 `UI 구조와 UX 원칙 참조`이며, `공식 UI 복제 지시서`가 아니다.

## 4. 기획 의도 반영 요약
- 첫 방문자는 10초 안에 `바로 시작`을 이해하고 눌러야 한다.
- 첫 판 종료 후 `재도전`이 가장 눈에 띄어야 한다.
- 모바일 세로 화면에서 보드, HUD, 조작부가 동시에 읽히고 눌려야 한다.
- Daily Challenge, 랭킹, 개인 최고 기록은 보조 동기이되 코어 플레이보다 앞에 나오면 안 된다.
- 광고/프로모션은 플레이와 첫 결과 CTA를 침범하면 안 된다.

## 4.1 기획자 개선의견 반영 요약
- 화면은 `읽히는 정보 UI`보다 `누르고 싶고 다시 오고 싶게 만드는 게임 서비스 UI`여야 한다.
- 독자 IP 전제이므로 공식 테트리스 트레이드드레스를 피하면서도, 자체 시각 언어와 감정 톤을 가져야 한다.
- 랜딩, 모드 선택, 결과, 랭킹은 정적 카드 반복이 아니라 `Hero`, `포스터`, `시상대`, `보상 오브젝트` 구조를 가져야 한다.
- Daily, 최근 성과, 랭킹, 공유는 단순 텍스트 정보가 아니라 `보상 언어`로 보이게 해야 한다.

## 5. 전역 UI 원칙
### 5.1 우선 레이아웃 기준
- 1차 기준 뷰포트는 `390x844`다.
- 2차 최소 보장 뷰포트는 `360x800`이다.
- 데스크톱 기준은 `1440x900`이다.

### 5.2 전역 여백 규칙
- 좌우 기본 화면 패딩은 모바일 `16px`, 데스크톱 `24px` 이상이다.
- 상하단 고정 UI는 safe area를 고려해 `padding-top`, `padding-bottom`에 시스템 inset을 더한다.
- 하단 조작부와 결과 CTA는 `env(safe-area-inset-bottom)`을 반영한 여백을 가져야 한다.

### 5.3 타이포그래피 기준
- Hero 수치 또는 최종 점수: `28~32px`, semibold 이상
- 화면 제목: `22~24px`
- 섹션 제목: `18~20px`
- 본문/설명: `15~17px`
- 보조 라벨: `12~13px`
- 작은 텍스트는 `11pt` 이하로 내리지 않는다. 이는 Apple 가이드를 따른다.

### 5.4 터치 타깃 기준
- 모든 주요 상호작용 요소는 최소 `48 x 48 CSS px` 이상으로 설계한다.
- 이는 Apple의 `44pt` 최소치와 Android의 `48dp` 권장치를 함께 만족시키기 위한 구현 기준이다.
- 회전, 이동, 드롭, Hold, 재도전 버튼은 최소보다 큰 `56 x 56` 이상을 권장한다.

### 5.5 색상과 대비 기준
- 작은 텍스트 대비는 최소 `4.5:1`
- 큰 텍스트 또는 강조 수치 대비는 최소 `3:1`
- 상태 구분은 색상만으로 하지 않는다.
- 기록 갱신, 플래그, 비활성 상태는 색 + 아이콘/텍스트/배지로 함께 표현한다.

### 5.6 시각 스타일 원칙
- 배경은 평면 흰색이나 디버그용 단색 배경이 아니라, 저채도 그리드/글로우/입자감을 활용한 게임형 분위기로 구성한다.
- 보드 영역은 가장 높은 대비를 가진 핵심 포커스 영역이어야 한다.
- 숫자와 CTA는 카드형 면 분리로 읽히게 한다.
- 개발 검증 문구, 내부 상태 설명, 임시 영문 카드 제목은 사용자 화면에 남기지 않는다.

### 5.7 게임 서비스형 아트 방향
- 전 화면은 `모바일 문서 서비스`가 아니라 `살아 있는 게임 서비스`처럼 보여야 한다.
- 이를 위해 아래 요소를 기본 축으로 사용한다.
  - 전용 컬러 시스템
  - Hero 장면
  - 감정 표현 오브젝트 또는 마스코트
  - 보상 오브젝트화된 배지/메달/스탬프/트로피
  - 상태에 따라 강하게 달라지는 카드 계층
  - 최소한의 모션 언어

### 5.8 독자 IP 시각 언어 기준
- 공식 테트리스 로고, 전용 색 배치, 제품 특유의 시각 문법을 복제하지 않는다.
- 대신 아래 세 요소를 가진 독자 IP 방향을 사용한다.
  - 서비스 도우미 역할의 마스코트 또는 오브젝트군
  - 챌린지, 랭킹, 보상에 반복 등장하는 배지 체계
  - 배경과 카드에서 반복되는 테마 패턴

### 5.9 권장 톤앤무드 방향 2안
#### A안: Neon Arcade Lab
- 방향: 네온 그리드, 전기 글로우, 실험실형 장난기
- 장점: 게임 리듬, 속도감, 기록 경쟁 표현이 강하다.
- 적합 화면: 플레이, 결과, 랭킹
- 핵심 오브젝트: 에너지 코어, 전광판 배지, 스파크 파티클

#### B안: Toy Workshop Quest
- 방향: 완구형 오브젝트, 스티커 보상, 수집장 같은 즐거움
- 장점: 캐주얼함, 데일리 재방문, 마스코트 친화성이 강하다.
- 적합 화면: 랜딩, Daily, 설정, 닉네임 등록
- 핵심 오브젝트: 스탬프 배지, 장난감 메달, 도우미 마스코트

### 5.10 카드 계층 규칙
- Hero 카드: 화면의 1등 감정 요소
- Reward 카드: Daily, 최근 성과, 개인 최고 기록, 보상
- Utility 카드: 설정, 설명, 보조 정보
- 세 계층은 같은 배경색과 같은 크기 언어를 쓰지 않는다.

### 5.11 모션 언어 최소 기준
- 랜딩 진입 시 Hero 오브젝트와 CTA가 순차적으로 등장해야 한다.
- 점수 상승, 콤보, 기록 갱신, 랭킹 진입은 미세한 팝/글로우/배지 등장으로 반응해야 한다.
- 결과 화면의 `재도전`은 hover/tap/press 상태가 가장 강하게 살아 있어야 한다.

### 5.12 시각 자산 소싱 원칙
- 게임형 오브젝트, 배경 타일, 시상대 오브젝트, 보상 소품은 다른 무료 사이트보다 `OpenGameArt`를 먼저 검토한다.
- `OpenGameArt` 안에서는 `CC0 -> OGA-BY -> CC-BY` 순으로 선택하고, `CC-BY-SA`, `GPL`, `LGPL`은 원칙적으로 피한다.
- 버튼 프레임, 패널, 입력 프롬프트는 `Kenney`를 2차 소스로 사용한다.
- 설정, 뒤로가기, 닫기, 공유 같은 유틸리티 아이콘은 `Heroicons` 또는 `Material Symbols`를 우선한다.
- 공식 게임 화면의 스크린샷, 로고, 전용 오브젝트를 잘라서 쓰지 않는다.

## 6. 전역 컴포넌트 규칙
### 6.1 상단 헤더
- 좌측: 서비스명 또는 모드명
- 우측: 설정, 음소거/진동 상태, 일시정지
- 랜딩에서는 헤더를 최소화하고, 플레이 중에는 기능형 헤더로 전환한다.

### 6.2 CTA 계층
- Primary CTA는 화면당 1개만 가장 강한 색과 면적을 가진다.
- Secondary CTA는 윤곽선 또는 저강도 톤을 쓴다.
- Destructive CTA는 색상과 문구로 명시 분리한다.

### 6.3 카드 규칙
- 카드 하나는 목적 하나만 가진다.
- 하나의 카드 안에 서로 다른 CTA를 3개 이상 넣지 않는다.
- 정보 카드와 행동 카드는 시각적으로 구분한다.

### 6.4 보상 언어 규칙
- `Daily Challenge`, `최근 성과`, `랭킹`, `공유`, `기록 저장`은 단순 텍스트 메뉴가 아니라 `획득`, `달성`, `도전`, `자랑` 감정으로 표현한다.
- 이를 위해 수치 외에 배지, 스탬프, 메달, 불꽃, 상위 퍼센트, 갱신 마크 같은 오브젝트 표현을 사용한다.

## 7. 화면별 UI 설계
## 7.1 SC-01 랜딩
### 목적
- 즉시 플레이를 가장 먼저 유도한다.

### 모바일 구조
1. 상단 얇은 헤더
   - 좌측 서비스명
   - 우측 설정 버튼
2. 메인 히어로
   - 큰 타이틀
   - 한 줄 설명
   - Hero 장면 또는 마스코트/오브젝트 쇼케이스
   - Primary CTA `바로 시작`
   - Secondary CTA `모드 선택`
   - Tertiary CTA `랭킹`
3. Daily Challenge 프리뷰 카드
   - 오늘 목표
   - 진행 상태
   - `도전하기`
4. 개인 최고 기록/최근 성과 카드
5. 공지 요약 1줄

### 배치 규칙
- 첫 화면에서 `바로 시작`, `모드 선택`, `랭킹` 3개가 모두 보여야 한다.
- 첫 화면의 가장 큰 면적은 히어로와 `바로 시작`이 가져야 한다.
- Daily Challenge는 히어로 아래에 두되, `바로 시작`보다 강하게 보이면 안 된다.
- `최근 성과`는 단순 숫자 문장이 아니라 배지, 상위 퍼센트, 갱신 마크 등 보상 언어로 표현한다.

### 금지 해석
- 기술 검증용 카드, API 상태 설명, 개발 단계 문구를 랜딩 첫 화면에 노출하지 않는다.
- 랜딩에서 스크롤 전에는 광고, 프로모션, 장문 설명을 배치하지 않는다.
- Hero 없는 카드 리스트형 첫 화면으로 마무리하지 않는다.

## 7.2 SC-02 모드 선택
### 목적
- 세 모드의 차이를 5초 내 이해시키고 선택하게 한다.

### 모바일 구조
- 상단: 뒤로가기, 제목 `모드 선택`
- 본문: 3개 모드 카드 세로 스택
  - 대표 비주얼 또는 포스터 장면
  - 모드명
  - 1줄 감정 카피
  - 핵심 목표
  - 경쟁 또는 보상 포인트
  - 종료 조건
  - `이 모드 시작`
- 하단: Daily Challenge 활성 시 강조 배너

### 카드 규칙
- Marathon: `오래 살아남아 고득점`
- Sprint: `40라인 최단 시간`
- Daily Challenge: `오늘의 목표와 보상`
- 세 카드의 배경, 아이콘 오브젝트, 감정 카피는 서로 다른 테마를 가져야 한다.

### 데스크톱 확장
- 3열 카드 배치 가능
- 카드 하단 CTA 정렬 고정

### 금지 해석
- 설명 카드 3개를 나열한 정보 목록처럼 보이게 만들지 않는다.
- `선택`과 `시작`을 이중 CTA로 분리해 행동 결정을 흐리게 만들지 않는다.

## 7.3 SC-03 게임 화면
### 목적
- 화면 내 모든 요소가 플레이를 돕고, 플레이를 방해하지 않아야 한다.

### 모바일 세로 기준 구조
1. 상단 상태 바
   - 좌측: 현재 모드
   - 중앙: 점수 또는 목표 수치
   - 우측: 일시정지
2. HUD 리본
   - 점수
   - 라인
   - 레벨 또는 목표 진행
   - 개인 최고 기록 대비 상태
3. 플레이 영역
   - 중앙: 보드
   - 보드 좌측 상단 바깥: Hold 카드
   - 보드 우측 상단 바깥: Next 5 카드 스택
4. 상태 피드백 영역
   - Combo
   - Back-to-Back
   - Perfect Clear
5. 하단 조작 도크
   - 왼쪽 군: `좌`, `우`
   - 중앙: `소프트 드롭`
   - 오른쪽 군: `회전`, `하드 드롭`, `Hold`

### 배치 규칙
- 보드는 항상 시각 중심축에 놓는다.
- Hold와 Next는 보드 양옆 상단에 붙여, 보드와 관계가 즉시 읽히게 한다.
- 점수/라인/레벨은 보드 위에, 조작 버튼은 보드 아래에 둔다.
- 조작 버튼은 스크롤 없이 항상 보여야 한다.
- 하단 조작 도크는 safe area 위에 고정한다.
- 게임 화면 전체는 모바일 웹뷰의 현재 높이 안에 완전히 들어와야 한다.
- `HUD + 보드 + 조작 도크` 때문에 화면 세로 높이가 웹화면 높이폭을 넘어서 아래쪽 일부를 못 보는 상태를 금지한다.
- 세로 높이가 부족할 경우 우선순위는 `보드 축소 -> HUD 밀도 축소 -> 패널 높이 축소` 순으로 조정하고, 조작 도크를 화면 밖으로 밀어내지 않는다.

### 시각 규칙
- Ghost는 실제 블록과 같은 색상 계열을 사용하되, 가능한 한 가장 연한 톤과 낮은 불투명도로 보인다.
- Active piece, ghost, locked blocks의 시각 위계가 분명해야 한다.
- 배경은 존재하되 보드 셀 대비를 해치면 안 된다.
- 점수, 콤보, 레벨은 상승 감각이 느껴지는 숫자 스타일과 미세 모션을 가진다.
- Next, Hold, 점수 패널은 같은 형태의 입력 패널처럼 보이면 안 된다.

### 접근성 규칙
- 고대비 모드에서는 셀 경계선과 HUD 수치를 더 강하게 표시한다.
- 조작 버튼은 아이콘만 두지 말고 텍스트 또는 툴팁 라벨을 함께 가진다.

### 금지 해석
- 디버그 정보, 세션 ID, bootstrap 상태, 개발 설명 텍스트를 플레이 영역에 노출하지 않는다.
- HUD와 조작부를 한 컬럼에 길게 쌓아 스크롤해야 보이는 구조를 금지한다.
- 게임 화면 하단이 웹뷰 밖으로 넘어가 사용자가 아래쪽 조작부나 정보 영역을 볼 수 없게 만드는 구조를 금지한다.
- 목표, 접근성, 설정 정보를 보드와 동일 강도의 상시 카드로 늘어놓지 않는다.
- 조작 버튼을 웹 폼 버튼처럼 밋밋하게 만들지 않는다.

## 7.4 SC-04 일시정지 오버레이
### 목적
- 플레이 재개를 최우선으로 제공한다.

### 구조
- 중앙 모달 또는 반투명 오버레이 카드
- 제목 `일시정지`
- Primary CTA `재개`
- Secondary CTA `설정`
- Tertiary CTA `나가기`

### 규칙
- `재개`가 가장 위 또는 가장 왼쪽 첫 버튼이어야 한다.
- 설정은 바텀시트나 내부 패널로 열고, 완전한 화면 전환을 우선하지 않는다.

## 7.5 SC-05 결과 화면
### 목적
- 기록 확인보다 재도전 행동을 더 빠르게 만든다.

### 모바일 구조
1. Hero 결과 카드
   - 최종 점수 또는 기록 시간
   - 제거 라인
   - 레벨
   - 최대 콤보
   - 랭킹 반영 상태
2. PB/성과 배지 줄
   - 개인 최고 기록 갱신
   - 첫 3판 강조 배지
   - Daily Challenge 성공/실패
3. CTA 그리드 2 x 2
   - `재도전`
   - `랭킹`
   - `공유`
   - `기록 저장`
4. 보조 정보
   - 다음 목표
   - 챌린지 보상

### 강조 규칙
- `재도전`이 시각적으로 가장 강해야 한다.
- 첫 3판 PB 갱신 시 Hero 카드 배경/배지/카피가 일반 결과와 분명히 달라야 한다.
- CTA 4개는 첫 화면에서 모두 보여야 한다.
- 결과 Hero는 `성과 화면`처럼 보여야 하며, 점수/기록/랭킹 중 최소 1개는 압도적인 1등 정보여야 한다.
- 공유 CTA는 자랑 욕구를 자극하는 시각 언어를 가져야 한다.

### 금지 해석
- 결과 카드 아래 긴 설명문을 먼저 넣고 CTA를 화면 하단으로 밀지 않는다.
- 첫 CTA 영역에는 광고, 프로모션, 실험 배너를 넣지 않는다.
- 동일한 크기의 박스 반복으로 결과 화면을 구성하지 않는다.

## 7.6 SC-06 랭킹 화면
### 목적
- 모드별 경쟁 구조를 빠르게 읽고 자신의 위치를 확인하게 한다.

### 구조
- 상단: 뒤로가기, 제목
- 1행 탭: `Marathon`, `Sprint`, `Daily`
- 2행 세그먼트: `일간`, `주간`, `전체`
- 내 기록 카드
- 랭킹 리스트
  - 순위
  - 닉네임
  - 기록값
  - 달성 시각
  - 본인 강조

### 규칙
- Marathon과 Daily는 점수형 숫자 강조
- Sprint는 시간형 숫자 강조
- 본인 행은 배경 강조로 바로 찾을 수 있어야 한다.
- Top 3는 일반 리스트와 구분되는 카드형 시상대 또는 메달 처리 구조를 가진다.
- 랭킹 탭은 일반 기능 탭보다 배지/모드 선택에 가깝게 보여야 한다.

## 7.7 SC-07 닉네임 등록 모달
### 목적
- 결과 흐름을 끊지 않고 기록 등록을 완료하게 한다.

### 구조
- 제목 `닉네임 등록`
- 입력 필드 1개
- 검증 규칙 요약
- Primary CTA `기록 저장`
- Secondary CTA `나중에`

### 규칙
- 전체 페이지 전환보다 바텀시트/모달을 우선한다.
- 금칙어, 길이 오류는 필드 하단 인라인 메시지로 표시한다.

## 7.8 SC-08 설정
### 목적
- 필수 옵션만 빠르게 조정하게 한다.

### 구조
- 사운드
- 진동
- 효과 강도
- Ghost Piece
- 고대비 모드
- 조작 가이드

### 규칙
- 모바일에서는 바텀시트 우선
- 토글, 세그먼트, 슬라이더만 사용하고 복잡한 폼을 만들지 않는다.
- 각 항목은 한 줄 설명을 가진다.

## 7.9 SC-09 Daily Challenge 상세
### 목적
- 오늘 목표와 보상을 읽고 바로 도전하게 한다.

### 구조
- 챌린지 제목
- 챌린지 비주얼 또는 오브젝트
- 목표 카드
- 제한 규칙
- 현재 진행도
- 보상
- Primary CTA `도전 시작`

### 규칙
- 목표 수치와 보상은 같은 카드 안에서 시각적으로 연결한다.
- 실패해도 재도전 가능 메시지를 명확히 남긴다.
- Daily는 단순 설명 시트가 아니라 `오늘의 보상 안내판`처럼 보여야 한다.

## 8. 반응형 전환 규칙
### 8.1 모바일 세로
- 한 화면 한 목표 원칙
- 보드 중심
- 하단 고정 조작 도크
- 게임 화면은 세로 스크롤 없이 한 웹화면 안에 완전 수용

### 8.2 태블릿/데스크톱
- 보드 중앙 유지
- 좌측 보조 정보, 우측 Hold/Next 또는 그 반대의 3단 레이아웃 허용
- 결과 화면은 좌측 Hero 카드, 우측 CTA/랭킹 상태 2열 배치 허용

## 9. 개발자 구현 금지사항
- 임시 영문 설명 카드, 개발 단계 라벨, 검증용 문장을 사용자 화면에 남기지 않는다.
- 랜딩에서 `바로 시작`보다 큰 보조 카드나 배너를 두지 않는다.
- 모바일에서 HUD, 보드, 조작부를 한 번에 못 보는 구조를 만들지 않는다.
- 모바일에서 게임 화면 하단 일부가 잘려 사용자가 아래쪽을 못 보는 상태를 허용하지 않는다.
- 결과 화면에서 `재도전`보다 `랭킹`, `공유`, `기록 저장`이 더 강하게 보이면 안 된다.
- settings를 독립 저우선 전용 페이지로만 빼서 플레이 중 접근 흐름을 길게 만들지 않는다.
- 화면 대부분을 동일 톤의 카드 박스로 반복하지 않는다.
- 플레이 화면을 설정 패널이나 관리 패널처럼 보이게 만들지 않는다.
- 라이선스 미확보 상태에서 공식 테트리스 트레이드드레스처럼 보이는 복제를 하지 않는다.

## 10. 개발자 인계 결론
- 본 서비스의 화면 UI는 `즉시 플레이`, `모바일 세로 우선`, `빠른 재도전`, `명확한 실력 체감`, `살아 있는 게임 서비스 톤`이라는 기획 의도에 맞춰 재정렬되어야 한다.
- 개발자는 기존 임시 화면 구성을 유지한 채 세부만 수정하면 안 된다.
- 화면에 쓰는 실제 게임형 오브젝트와 배경 타일은 `OpenGameArt` 1순위, `Kenney` 2순위, `Game-icons/Heroicons/Material Symbols` 보조 원칙으로 조합한다.
- 구현 순서는 `랜딩 -> 게임 화면 -> 결과 화면 -> 모드 선택 -> 랭킹 -> 설정/닉네임/Daily 상세` 순으로 가져가는 것을 권장한다.
