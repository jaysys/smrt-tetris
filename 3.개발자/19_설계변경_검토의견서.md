# 설계변경 검토의견서

## 1. 목적
- 본 문서는 `2.설계문서/18_개발자전달_설계변경사항.md`, `2.설계문서/20_개발자전달_게임엔진구현지시서.md`, `2.설계문서/21_개발자전달_추가설계변경사항.md`를 검토한 뒤, 현재 개발 상태와의 정합성, 개발 영향, 반영 우선순위를 정리하기 위한 의견서다.

## 2. 검토 결론
- 설계변경 방향은 전체적으로 타당하다.
- 변경사항은 구현 가능하다.
- 다만 영향 성격이 서로 다르므로 아래처럼 나눠 관리해야 한다.
  - 즉시 선행 보정 필요
  - 후속 단계 반영 예약
  - 닫힌 단계 재오픈 없이 기준만 갱신

내 의견은 다음과 같다.
- 이미 제3자 테스트 `Pass`로 종료된 D1은 재오픈하지 않는다.
- 대신 D2 착수 전에 설계변경 중 현재 코드와 직접 충돌하는 항목을 선행 보정 작업으로 등록한다.
- 특히 엔진 규칙, 분석 이벤트, 모바일 세로, 접근성 최소 옵션은 후순위로 미루면 안 된다.

## 3. 항목별 검토 의견
### 3.1 모바일 세로 우선순위 상향
- 의견: 타당하다.
- 현재 상태: 부분 충족이다.
- 근거:
  - 공개 웹은 반응형 미디어쿼리를 이미 갖고 있다: `apps/web/src/app/globals.css`
  - 그러나 현재 레이아웃은 범용 반응형 수준이며, `360x800` 기준 HUD/조작부 완료 조건이 문서상 닫혀 있지 않다.
- 판단:
  - D1 결과를 뒤집을 사유는 아니다.
  - 하지만 D2 시작 전에 모바일 세로 기준을 명시적으로 통과 조건으로 재등록해야 한다.

### 3.2 분석 이벤트 5종 최소 세트 고정
- 의견: 즉시 반영이 필요하다.
- 현재 상태: 미충족이다.
- 근거:
  - 분석 이벤트 타입 패키지는 존재한다: `packages/analytics/src/index.ts`
  - 현재 앱 코드 전체 기준으로 `landing_view`, `quick_start_click`, `game_start`, `game_finish`, `retry_click` 이벤트 이름 연결이 확인되지 않는다.
- 판단:
  - 이 항목은 더 이상 D6 후반 점검 항목이 아니다.
  - D2 착수 전 또는 D2 초반에 반드시 선행 구현해야 한다.

### 3.3 접근성 최소 옵션 선반영
- 의견: 타당하다.
- 현재 상태: 부분 충족이다.
- 근거:
  - 설정 데이터에는 `highContrastMode`가 존재한다.
  - 하지만 실제 UI에 고대비 모드, 대체 조작, 명시적 접근성 상태 반영은 아직 없다: `apps/web/src/app/game-client.tsx`, `apps/web/src/app/globals.css`
- 판단:
  - 설계변경을 기준으로 보면 접근성 최소 옵션은 D6가 아니라 D2~D3 이전 선행 기준으로 옮겨야 한다.

### 3.4 Marathon 랭킹 동점 규칙 확정
- 의견: 타당하다.
- 현재 상태: 미구현이므로 충돌은 아직 없다.
- 판단:
  - D4 랭킹 구현 전에 API 정렬 규칙, DB 조회 정렬, 테스트 데이터 규칙을 이 기준으로 잠가야 한다.
  - 지금 바로 필요한 것은 구현 변경이 아니라 계약 고정이다.

### 3.5 첫 3판 개인 최고 기록 강조 연출
- 의견: 타당하다.
- 현재 상태: 미충족이다.
- 근거:
  - 현재 결과 화면은 단일 상태 결과 카드다: `apps/web/src/app/game-client.tsx`
  - 초기 3판 PB 강조 상태, 축하 카피, 배지, CTA 인접 유도 요소는 아직 없다.
- 판단:
  - 결과 화면 컴포넌트 상태를 하나 더 늘려야 한다.
  - 기록 루프와 연결되므로 D3 이전에 UI 상태 모델을 먼저 준비하는 편이 맞다.

### 3.6 광고 금지 구역 규칙
- 의견: 타당하다.
- 현재 상태: 현재 구현과 직접 충돌은 없다.
- 판단:
  - 현재는 광고 슬롯 구현이 없으므로 즉시 수정 필요는 없다.
  - 다만 결과 화면 첫 CTA 주변은 향후 프로모션 슬롯 금지 영역으로 예약해야 한다.

### 3.7 복귀 루프 방향성
- 의견: 타당하다.
- 현재 상태: 미구현이나 문제는 없다.
- 판단:
  - 즉시 구현 대상은 아니다.
  - 다만 이벤트와 결과 메타데이터 구조를 닫아버리면 나중에 확장이 불편해진다.

### 3.8 출석 보상과 연속 방문 구분
- 의견: 타당하다.
- 현재 상태: 아직 해당 기능이 없으므로 충돌은 없다.
- 판단:
  - D5 보상/업적 설계 시 두 개념을 별도 카운터로 분리해야 한다.

## 4. 엔진 규칙 변경에 대한 별도 의견
### 4.1 이미 충족한 부분
- `7-bag` 공급기: 충족
- 가시 `10x20` + 숨김 `2행`: 충족
- 스폰 시 `0` 상태 시작: 충족
- Hold 기본 제공, 1턴당 1회 제한: 충족
- Hold 재호출 시 스폰 방향 재정렬: 충족
- Ghost 표시 제공: 충족

근거:
- `7-bag`: `packages/game-engine/src/core/bag.ts`
- 스폰/보드/회전 상태: `packages/game-engine/src/core/board.ts`, `packages/game-engine/src/core/constants.ts`
- Hold/Ghost: `packages/game-engine/src/core/session.ts`

### 4.2 즉시 보정이 필요한 부분
- 전이별 SRS 벽차기 표 미충족
- 5개 후보 순서 기반 회전 테스트 미충족
- 판정 순서 전체 정합화 미충족
- 자연 낙하 중력값과 소프트 드롭 15배 규칙 미구현

근거:
- 현재 회전 구현은 `from-rotation -> to-rotation`별 표를 쓰지 않고, 블록 타입별 공통 배열만 사용한다: `packages/game-engine/src/core/board.ts`
- 현재 `WALL_KICKS.normal`, `WALL_KICKS.I`는 전이별 표가 아니라 단일 배열이다: `packages/game-engine/src/core/constants.ts`
- 현재 점수 계산은 라인 수 기준 기본 점수만 더하며, 착지 -> 고정 -> 라인 수 계산 -> Perfect Clear -> Back-to-Back -> Combo -> 점수 계산 -> HUD 갱신 -> 다음 블록 스폰 -> 스폰 실패 판정의 전체 시퀀스가 아직 구현 기준으로 고정되어 있지 않다: `packages/game-engine/src/core/session.ts`
- 현재 낙하는 고정 interval과 단순 soft drop 점수 증가 방식이며, 설계 문서의 레벨별 중력/15배 가속 모델과 다르다: `apps/web/src/app/game-client.tsx`, `packages/game-engine/src/core/session.ts`

### 4.3 판단
- 이 엔진 규칙 차이는 단순 개선이 아니라 설계와 직접 충돌하는 항목이다.
- 따라서 D2 기능을 더 얹기 전에 엔진 보정 작업을 먼저 넣는 것이 맞다.
- 특히 `Marathon`, `Sprint`, `Daily Challenge`를 모드별로 확장하기 전에 엔진 규칙을 먼저 고정해야 이후 테스트 재작업을 줄일 수 있다.

## 7. 최종 의견
- 이번 설계변경은 수용하는 것이 맞다.
- 다만 모든 변경을 동일한 우선순위로 다루면 안 된다.
- 현재 기준에서 가장 중요한 것은 `엔진 규칙 보정`, `분석 이벤트`, `모바일 세로`, `접근성 최소 옵션`이다.
- 이미 종료된 D1을 재오픈할 필요는 없지만, D2는 기존 계획 그대로 바로 들어가면 안 된다.
- D2는 최신 설계변경을 반영한 선행 보정 작업을 먼저 등록한 뒤 착수하는 것이 타당하다.
