## 3. 반영 완료 항목
### 3.1 엔드포인트별 필수 검증 필드 구분
- 공통 검증 필드와 결과 제출 API 전용 검증 필드를 분리했다.
- 랭킹, Daily Challenge, 닉네임, 운영 API도 각각 필수 검증 필드를 구분해 명시했다.

### 3.2 테스트 전용 제어 수단의 표준 경로
- `TEST_MODE=true` 환경에서만 테스트 제어 기능이 열리도록 요구사항을 명시했다.
- 명령형 표준 경로는 아래 기준으로 정리했다.

```bash
pnpm db:reset
pnpm db:migrate
pnpm db:seed
pnpm test:e2e
```

- API형 표준 경로는 `/v1/test/*` 또는 동등한 테스트 전용 엔드포인트로 정리했다.

### 3.3 실패 산출물 저장 규약
- 로컬/CI 공통 저장 경로 규칙을 추가했다.
- 파일명 규약과 최소 보존 범위를 명시했다.
- 최소 보존 대상은 `P0` 실패 케이스 전체로 정의했다.

### 3.4 게임 자동화 지원 방식 우선순위
- 개발팀 제안 우선순위에 동의한다.
- 적용 우선순위는 아래와 같이 정리했다.
  1. 시드 고정 + 입력 재생
  2. 테스트 전용 디버그 훅
  3. 테스트 전용 상태 주입 API

### 3.5 `pnpm install` 인수 기준의 환경 전제
- 외부 네트워크 접근 가능 환경 또는 사내 패키지 미러 환경을 전제로 명시했다.
- 현재 워크스페이스의 DNS 제한 상태를 고려해, 설치 성공 자체를 코드 불량과 동일시하지 않도록 전제를 문서에 반영했다.

## 4. 테스터 입장 추가 확인 사항
- 개발팀이 반영한 자동화 대응 기준은 Must 품질 게이트로 유지되어야 한다.
- 특히 아래 항목은 릴리스 직전이 아니라 각 단계 완료 조건에 포함되어야 한다.
  - `data-testid` 제공
  - 테스트 시계 주입
  - 멱등성 재현
  - 실패 산출물 수집
  - `pnpm test:e2e` 실행 가능 상태

## 5. 현재 합의 상태
- 자동화 대응 요구사항의 핵심 취지는 개발팀과 테스터가 동일하게 수용했다.
- 현재 남은 과제는 요구사항 조정이 아니라 실제 코드 구현과 명령 검증이다.
- 따라서 이후 단계에서는 문서 논의보다 다음 항목의 구현 여부를 기준으로 확인한다.
  - 실행 명령 동작
  - 테스트 전용 게이트 제공
  - P0 E2E 자동화 통과
  - 실패 산출물 저장 확인
