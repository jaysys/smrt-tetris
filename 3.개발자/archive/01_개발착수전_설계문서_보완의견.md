# 개발착수전 설계문서 보완 의견

## 1. 문서 목적
본 문서는 `2.설계문서`를 재검토한 결과를 기록한다. 초기 보완 의견이 반영된 이후 현재 개발 착수 가능 수준을 다시 판단하고, 아직 미진하거나 문서 간 충돌이 남아 있는 항목을 추가 의견으로 정리한다.

## 2. 재검토 결론
현재 설계문서는 최초 검토 시점보다 상당히 보강되었으며, MVP 개발 착수 가능성은 높아졌다.

특히 다음 항목은 의미 있게 개선되었다.
- 시간 기준이 기능설계서, 시스템설계서, API명세서, DB설계서에 반영되었다.
- 게임 규칙 수치테이블이 별도 문서로 분리되었다.
- 테스트데이터 명세서, QA실행가이드, API 예시응답 문서가 추가되었다.
- 결과 제출 멱등성과 중복 방지 정책이 시스템설계서, API명세서, DB설계서에 반영되었다.
- 부정행위 검증 기준이 정량 규칙으로 일부 구체화되었다.
- 운영배포체크리스트에 환경변수, 표준 명령, 로그 확인 위치가 추가되었다.

다만 현재 상태를 `바로 구현 가능한 설계 문서`로 보더라도, 아래 잔여 이슈는 반드시 추가 보완하는 것이 바람직하다. 이 항목들은 구현 충돌, 테스트 재현 불가, 문서 간 해석 차이를 유발할 가능성이 있다.

## 3. 반영 완료로 판단하는 항목
### 3.1 시간 기준 통일
- UTC 저장, KST 표시, KST 기준 Daily/Weekly 집계 규칙이 문서 전반에 반영되었다.
- 날짜 전환 테스트 기준도 별도 문서로 분리되어 있다.

### 3.2 테스트 독립성 보강
- 테스트 URL, 기본 계정, 시드 데이터, DB 초기화 절차, 날짜 전환/네트워크 장애 테스트 절차가 추가되었다.
- 외부 테스터가 개발자 도움 없이 절차를 따라갈 수 있는 수준으로 개선되었다.

### 3.3 API 계약 보강
- 핵심 API에 대해 성공/실패 status, 재시도 정책, 멱등성 규칙, 중복 방지 정책이 추가되었다.
- 예시 응답 문서가 별도로 생겨 프런트엔드, 백엔드, QA 간 응답 해석 기준이 좋아졌다.

### 3.4 운영 런북 보강
- 환경변수, 표준 명령, 스모크 테스트 URL, 로그 확인 위치, 우선 확인 API가 정의되었다.
- 초기 운영 점검 관점에서 필요한 문서 수준은 크게 향상되었다.

## 4. 추가 보완이 필요한 항목
### 4.1 DB 기술 선택과 DB설계서 타입 체계가 아직 충돌한다
현재 시스템설계서는 SQLite3를 기준으로 전환되었으나, DB설계서는 여전히 PostgreSQL 성격의 타입과 표현을 유지하고 있다.

대표 예시:
- `uuid`
- `jsonb`
- `timestamptz`
- `bigserial`

문제점:
- 구현자가 SQLite 기준 스키마를 작성할 때 실제 컬럼 타입과 문서가 어긋난다.
- 마이그레이션 작성, ORM 설정, 인덱스 전략, PK 생성 방식에서 혼선이 생긴다.
- 제3자 테스터가 DB를 직접 점검할 경우 문서와 실제 스키마가 다를 가능성이 높다.

추가 반영 필요:
- SQLite 기준 실제 저장 타입 정의
- UUID 생성 전략
- JSON 컬럼 저장 방식
- 시간 컬럼 저장 형식
- PostgreSQL 전환 시 변경 예정 항목과 현재 MVP 기준 항목의 구분

### 4.2 게임규칙 수치테이블이 아직 엔진 구현 기준으로 완전히 닫히지 않았다
별도 수치테이블 문서는 추가되었으나, 몇몇 핵심 규칙은 여전히 해석 여지가 있다.

대표 미진점:
- 스폰 X/Y가 `중앙 정렬 기준`, `버퍼 시작 행`처럼 상대 표현이다.
- 블록별 초기 회전 상태가 없다.
- 블록별 실제 셀 매트릭스 정의가 없다.
- 벽차기 규칙이 회전 방향별 상태 전이 기준 없이 요약돼 있다.
- 일반 블록과 장축 블록만 있고 O 블록의 예외 규칙이 없다.

문제점:
- 프런트엔드 게임 엔진 구현이 개발자별로 달라질 수 있다.
- 서버 검증 로직이 클라이언트와 동일 판정을 재현하기 어렵다.
- 벽 근처 회전, 스폰 실패, 특수 블록 처리 테스트가 흔들린다.

추가 반영 필요:
- 블록별 초기 좌표와 초기 회전 상태
- 블록별 셀 매트릭스
- 시계/반시계 회전 전이표
- 블록 종류별 벽차기 상세표
- O 블록 예외 처리 기준

### 4.3 API명세서가 핵심 API 중심으로만 상세하고 나머지 엔드포인트는 아직 비대칭적이다
핵심 흐름인 `game-sessions`와 `finish`는 많이 개선되었지만, 나머지 엔드포인트는 아직 필드 수준 계약이 부족하다.

대표 미진점:
- `GET /me/stats`
- `GET /settings`
- `GET /daily-challenge`
- `GET /missions`
- `GET /share-links/{shareId}`
- 운영 API 전반

문제점:
- 프런트엔드 화면 구현 시 필요한 응답 구조를 각자 추정하게 된다.
- QA가 응답 필드 누락을 사양 위반으로 판정하기 어렵다.
- 운영 백오피스 구현과 공개 화면 구현의 계약 완성도가 다르다.

추가 반영 필요:
- 모든 엔드포인트에 대해 요청/응답 필드 표 추가
- enum, null 허용 여부, 배열 구조, 중첩 객체 구조 명시
- 운영 API의 성공/실패 응답 예시 보강

### 4.4 QA실행가이드의 날짜 전환 테스트 절차가 아직 실행 방법까지는 닫혀 있지 않다
QA실행가이드는 절차를 제시하지만, 실제로 시간을 어떻게 제어할지까지는 명시하지 않는다.

대표 미진점:
- `테스트 시각을 KST 23:59:50로 맞춘다`는 문장만 있고 실현 방법이 없다.
- 시스템 시간 변경, 환경변수 주입, 테스트용 시계 mock 중 어느 방식을 쓰는지 없다.
- `일간 랭킹 period key 변경 여부 확인`이라고 되어 있으나 이를 어떤 API/화면/DB에서 확인하는지 명확하지 않다.

문제점:
- 제3자 테스터가 같은 날짜 전환 테스트를 재현하기 어렵다.
- 로컬 테스트와 CI 테스트 방법이 달라질 수 있다.

추가 반영 필요:
- 시간 제어 방식 1개를 표준으로 확정
- 날짜 전환 확인 대상 API 또는 관리자 화면 명시
- 목킹 도구 또는 플래그 사용 여부 명시

### 4.5 운영 명령은 추가되었지만 실제 저장소 스크립트 존재 여부와 연결돼야 한다
운영배포체크리스트에 표준 명령과 롤백 예시가 추가된 점은 좋다. 다만 이것이 실제 저장소 스크립트와 1:1로 대응되는지는 아직 문서만으로 보장되지 않는다.

대표 미진점:
- `pnpm db:migrate:rollback`
- `pnpm stop`
- `pnpm start`

문제점:
- 운영 문서가 실제 프로젝트 명령과 다르면 배포/복구 시 오류가 난다.
- 외부 테스터 또는 신규 개발자가 문서를 그대로 따라 했을 때 실패할 수 있다.

추가 반영 필요:
- 실제 `package.json` 스크립트명과 문서 일치 여부 확인
- 명령이 아직 없다면 추후 구현 예정임을 명시
- 환경별 실행 명령 차이 명시

## 5. 권장 수정 우선순위
### 우선순위 높음
- DB설계서를 SQLite 기준 타입 체계로 정리
- 게임규칙 수치테이블에 블록별 스폰/회전/매트릭스 정의 보강
- API명세서의 비핵심 엔드포인트 상세 계약 보강

### 우선순위 중간
- QA 날짜 전환 테스트의 실제 시간 제어 방법 확정
- 운영 체크리스트의 명령과 실제 저장소 스크립트 정합성 확인

## 6. 최종 판단
현재 설계문서는 초기 상태와 비교하면 개발 착수 가능 수준으로 충분히 개선되었다. 특히 테스트 독립성, 시간 기준, 멱등성, 운영 체크리스트 측면의 보강은 적절하다.

다만 설계 완성도를 한 단계 더 높이려면 `DB 기술 정합성`, `게임 엔진의 절대 규칙 정의`, `비핵심 API 계약 완결성`, `실행 가능한 QA 시간 제어 방법`을 추가 보완해야 한다. 이 네 가지가 정리되면 프런트엔드와 백엔드 구현 편차를 더욱 줄일 수 있고, 제3자 테스터도 문서만으로 안정적으로 검증할 수 있다.
