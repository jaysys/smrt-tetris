# QA실행가이드

## 1. 목적
외부 테스터가 개발자 도움 없이 로컬 또는 테스트 환경을 세팅하고 테스트를 수행할 수 있도록 절차를 제공한다.

## 2. 로컬 실행 절차
```bash
pnpm install
pnpm db:migrate
pnpm db:seed
pnpm dev
```

접속 URL
- 웹: `http://localhost:3000`
- API: `http://localhost:3001/v1`
- 운영: `http://localhost:3000/admin`

## 3. 테스트 수행 순서
1. 랜딩과 즉시 플레이
2. 코어 조작과 점수/라인/게임 오버
3. 모바일 세로 레이아웃과 접근성 최소 옵션
4. 결과 화면과 재도전
5. 핵심 분석 이벤트 적재
6. 모드별 규칙
7. 닉네임/개인기록/랭킹
8. Daily Challenge/미션/업적
9. 광고 금지 구역과 코어 루프 보호 규칙
8. 운영 백오피스
9. 부정행위/오류/재시도

## 4. DB 초기화 및 재시드
```bash
pnpm db:reset
pnpm db:migrate
pnpm db:seed
```

## 5. 날짜 전환 테스트 방법
### 표준 시간 제어 방식
- 날짜 전환 테스트는 `서버 주입 시계(Test Clock)` 방식 하나로 통일한다.
- 표준 제어 수단은 `TEST_CLOCK_ISO_UTC` 환경변수 하나만 사용한다.
- 관리자 테스트 시계 설정 API는 사용하지 않는다. 필요 시 향후 비표준 보조 수단으로만 검토한다.
- 로컬 QA와 자동화 QA는 모두 같은 주입 시계 기준을 사용한다.

### 수행 절차
1. 서버는 기본적으로 UTC로 실행한다.
2. 테스트 시계 값을 `2026-03-11T14:59:50Z`로 설정한다. 이는 KST `2026-03-11 23:59:50`에 해당한다.
3. `GET /v1/daily-challenge` 결과의 `challengeId`, `validTo`를 기록한다.
4. `GET /v1/leaderboards?mode=MARATHON&period=daily` 결과의 `period`와 엔트리를 기록한다.
5. 테스트 시계 값을 `2026-03-11T15:00:00Z`로 변경한다. 이는 KST 자정이다.
6. 같은 API를 다시 호출한다.
7. `challengeId` 변경 여부와 일간 period 경계 변경 여부를 확인한다.

### 표준 확인 지점
- 1차 확인 API: `GET /v1/daily-challenge`
- 2차 확인 API: `GET /v1/leaderboards?mode=MARATHON&period=daily`
- 3차 확인 화면: 운영자 챌린지 화면
- 4차 확인 데이터: `daily_challenges`, `leaderboard_entries.period_key`

## 6. 네트워크 장애 테스트 방법
### 결과 제출 실패 후 재시도
1. devtools 네트워크를 `Offline`으로 전환한다.
2. 게임 종료 후 결과 제출을 시도한다.
3. 오류 상태 배지를 확인한다.
4. 네트워크를 복구한다.
5. 동일 요청이 멱등하게 1회만 승인되는지 확인한다.

### 랭킹 조회 실패
1. API 목킹 또는 프록시로 `/leaderboards`에 500 응답을 주입한다.
2. 랭킹 화면 오류 메시지를 확인한다.
3. 복구 후 재조회한다.

## 7. 운영 테스트 절차
1. 운영자 로그인
2. Daily Challenge 예약 생성
3. 공지 등록
4. 의심 기록 조회
5. 플래그 조치 수행

## 8. 결과 보고 형식
| 항목 | 기재 내용 |
|---|---|
| 테스트 일시 | YYYY-MM-DD HH:mm:ss KST |
| 환경 | 로컬/스테이징, 브라우저, OS |
| 테스트 케이스 ID | 예: TC-017 |
| 수행 결과 | Pass / Fail |
| 재현 절차 | 번호형 절차 |
| 기대결과 대비 실제결과 | 차이 명시 |
| 첨부 | 스크린샷, 영상, 로그 |

## 9. 자동화 QA 기준
- 자동화 테스트도 `TEST_CLOCK_ISO_UTC`를 통해 시간을 주입한다.
- 자동화는 브라우저 로컬 시간 조작을 사용하지 않는다.
- 시간 관련 판정은 모두 서버 응답과 DB 상태 기준으로 확인한다.
- 코어 자동화는 최소 `landing_view`, `quick_start_click`, `game_start`, `game_finish`, `retry_click` 적재 여부까지 함께 확인한다.
