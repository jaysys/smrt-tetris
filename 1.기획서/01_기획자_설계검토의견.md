# 기획자 설계검토의견

## 1. 검토 목적
- 본 문서는 `1.기획서/테트리스_웹서비스_요구사항정의서.md` 기준으로 `2.설계문서`가 기획 의도와 충분히 일치하는지 검토한 결과다.
- 수신 대상은 설계자이며, 목적은 `무엇이 잘 맞았고 무엇을 보강해야 하는지`를 명확히 전달하는 것이다.
- 검토 기준일: 2026-03-11

## 2. 총평

현재 설계는 기획 의도와 `대체로 잘 일치`한다. 특히 서비스의 핵심 성공요인인 `게스트 즉시 플레이`, `짧은 재도전 루프`, `Marathon/Sprint/Daily Challenge의 3축`, `랭킹과 개인 최고 기록`, `모바일 웹 대응`, `운영 가능성`, `IP 리스크 전제`가 전반적으로 잘 반영되어 있다.

기획자 판단으로는 현재 설계를 `개발 착수 가능한 수준`으로 본다. 다만 출시 경쟁력을 높이려면 아래 보완 사항은 설계 단계에서 더 명확히 고정할 필요가 있다. 결론은 `조건부 적합`이다.

## 3. 기획 의도와 잘 맞는 부분

### 3.1 핵심 진입 구조가 맞다
- 랜딩에서 `바로 시작`, `모드 선택`, `랭킹`의 3개 진입점을 유지한 점은 좋다.
- 로그인 없는 첫 진입을 기본으로 둔 점도 기획 의도와 정확히 맞다.
- 첫 플레이 전 튜토리얼을 짧고 스킵 가능하게 둔 점도 적절하다.

### 3.2 코어 플레이 루프가 잘 잡혀 있다
- `랜딩 -> 플레이 -> 결과 -> 재도전` 루프가 설계 중심에 놓여 있다.
- 결과 화면에서 `재도전`을 첫 CTA로 둔 점은 매우 중요하며, 기획 의도와 일치한다.
- 결과 화면 1초 이내 노출, 재도전 2초 이내 재시작 목표도 방향이 좋다.

### 3.3 경쟁과 재방문 구조가 살아 있다
- MVP에 `Marathon`, `Sprint`, `Daily Challenge`를 모두 넣은 점이 적절하다.
- 일간/주간/전체 랭킹, 개인 최고 기록, 닉네임 등록 구조도 코어 경쟁 루프와 맞다.
- Daily Challenge, 반복 미션, 업적, 운영 백오피스를 모두 엮은 점은 장기 잔존 전략과 부합한다.

### 3.4 출시 리스크 관점의 방어가 좋다
- 독자 IP 전제를 문서 상단에 명시한 점이 좋다.
- 일반 표현 사용, 법무 검토 전 브랜딩 유보, 부정행위 검증 구조, 운영자 제재 기능이 잘 연결되어 있다.
- 서버 검증과 랭킹 오염 방지 정책도 서비스 신뢰 확보 측면에서 적절하다.

## 4. 설계자에게 수정 요청하는 핵심 보완사항

### 4.1 모바일/접근성/분석이 일정상 너무 뒤로 밀려 있다
- 현재 MVP 개발계획서에서는 `모바일 세로 최적화`, `접근성 옵션`, `분석 이벤트 수집`이 Phase 5에 배치되어 있다.
- 하지만 기획 의도상 이 셋은 출시 직전 장식이 아니라 `초기 성공요인`에 직접 연결되는 필수 품질이다.
- 특히 이 서비스는 모바일 웹 유입이 핵심이므로, 모바일 세로 UX가 뒤로 밀리면 실제 MVP 품질이 흔들린다.

수정 요청
- 모바일 세로 기본 레이아웃과 핵심 HUD/조작부 검증은 최소 Phase 1~2 완료 조건으로 당겨야 한다.
- 접근성 최소 옵션(고대비, 시각 효과 축소, HUD 가독성)은 Phase 3 이전에 구현 범위를 고정해야 한다.
- 분석 이벤트는 출시 직전 부착이 아니라 `landing_view`, `quick_start_click`, `game_start`, `game_finish`, `retry_click`까지는 코어 루프 구현과 함께 붙여야 한다.

### 4.2 Marathon 동점 처리 기준 문구가 모호하다
- 기능설계서의 Marathon 랭킹 기준 문구는 해석 여지가 있다.
- `동점 시 생존 시간 짧은 순 아님, 더 적은 총 입력 수 우선`은 개발자, QA, 운영자가 동일하게 해석하기 어렵다.

수정 요청
- 랭킹 동점 처리 순서를 완전한 결정 규칙으로 다시 써야 한다.
- 예: `점수 내림차순 -> 총 생존 시간 오름차순/내림차순 중 택1 -> 총 입력 수 오름차순 -> 먼저 달성한 기록 우선`처럼 명시해야 한다.

### 4.3 초반 성취감 설계가 아직 약하다
- 기획 의도에서는 `첫 3판 동안 개인 최고 기록 갱신 연출 강화`가 중요했다.
- 현재 설계는 개인 최고 기록 노출은 있으나, `초기 3판에서 어떤 연출을 어떻게 강화할지`가 제품 설계 수준으로 내려오지 않았다.

수정 요청
- 첫 3판 동안은 개인 최고 기록 갱신 시 일반 결과와 다른 강조 연출 규칙을 명시해야 한다.
- 예: 결과 카드 시각 강조, 축하 카피, 보상 포인트 추가, 재도전 CTA와의 결합 방식을 구체화해야 한다.

### 4.4 복귀 루프는 데이터 추출만 있고 사용자 경험 설계가 없다
- 시스템설계에는 `Dormant Player Segment`가 있으나, 사용자가 실제로 어떤 복귀 경험을 받는지는 설계 문서에 없다.
- 현재 단계에서는 치명적 누락은 아니지만, 서비스 성장을 생각하면 기획 의도 대비 한 단계 부족하다.

수정 요청
- 최소한 후속 범위라도 `복귀 사용자에게 무엇을 보여줄지`를 설계 오픈 이슈로 더 구체화해야 한다.
- 예: 복귀 배너, 복귀 미션, 최근 최고 기록 회상, Daily Challenge 재진입 유도 중 어떤 방식으로 갈지 선택지를 남겨야 한다.

### 4.5 향후 수익화 삽입 지점 보호 규칙을 더 강하게 못 박아야 한다
- 현재 문서들은 `광고/결제는 MVP 제외`, `과금이 공정성을 훼손하면 안 됨` 수준까지는 잘 정리되어 있다.
- 다만 향후 실제 운영 단계에서 어느 화면에 광고를 넣어도 되는지, 어디는 절대 금지인지가 아직 제품 규칙으로 고정되어 있지는 않다.

수정 요청
- 설계 단계에서 광고 허용 위치와 금지 위치를 분명히 선언해야 한다.
- 최소한 `플레이 도중 광고 금지`, `첫 결과 화면의 첫 CTA 영역 침범 금지`, `재도전 직전 강제 광고 금지`는 제품 규칙으로 명시해야 한다.

## 5. 권장 추가 보강 항목

아래는 즉시 수정이 꼭 필요한 blocker는 아니지만, 기획 완성도를 높이는 데 유효하다.

- 출석과 연속 방문의 차이를 더 명확히 나눠 표현하면 좋다.
- 공유 결과 페이지에서 `상위 몇 %인지`를 더 전면에 보여주면 경쟁 유도가 강해진다.
- Daily Challenge 난이도 운영 원칙을 더 적어 두면, 운영자가 너무 쉬운 날과 너무 어려운 날을 반복하는 문제를 줄일 수 있다.
- 모바일 조작 버튼 배치에 대한 좌우 손잡이 옵션은 2차라도 설계 오픈 이슈로 남겨 두는 것이 좋다.

## 6. 최종 판정

### 6.1 판정
- `조건부 적합`

### 6.2 이유
- 핵심 기획 의도는 대부분 반영되어 있다.
- 특히 MVP의 본질인 `즉시 플레이`, `짧은 재도전`, `기록 경쟁`, `데일리 재방문`, `운영 가능성`은 설계에 잘 살아 있다.
- 다만 `모바일/접근성/분석의 일정 우선순위`, `랭킹 동점 규칙의 모호성`, `초기 성취감 강화`, `복귀 루프`, `향후 수익화 보호 규칙`은 설계자의 추가 정리가 필요하다.

### 6.3 설계자 액션 요청
1. MVP개발계획서에서 모바일/접근성/분석의 최소 필수 범위를 앞당겨 재배치할 것
2. Marathon 동점 처리 규칙을 단일 해석 가능 문구로 확정할 것
3. 첫 3판 개인 최고 기록 강화 연출 규칙을 기능설계서에 추가할 것
4. 복귀 사용자 경험을 오픈 이슈가 아닌 방향성 있는 후속 설계 항목으로 정리할 것
5. 향후 광고/수익화 금지 구역을 설계 문서에 명시할 것

## 7. 기획자 최종 의견 한 줄

이 설계는 방향이 맞다. 다만 `모바일 우선`, `초반 성취감`, `정확한 경쟁 규칙`, `코어 루프 보호 원칙`만 더 분명히 고정해야 시장에서 이기는 제품이 된다.

## 8. 설계자 회신 검토 결과

### 8.1 검토 대상

- 검토 기준일: 2026-03-11

### 8.2 기획자 판단
- 설계자의 회신은 `대체로 수용 가능`하다.
- 핵심 지적 5건에 대해 설계자가 문제 인식을 명확히 공유했고, 그중 4건은 즉시 반영, 1건은 후속 범위 명시 강화로 정리한 점이 타당하다.
- 따라서 기획자 입장에서도 `조건부 적합` 판단은 유지하되, 회신 방향 자체는 수용한다.

### 8.3 회신 수용 항목

- 모바일/접근성/분석을 MVP 핵심 품질로 보고 일정 우선순위를 앞당기겠다는 회신은 적절하다.
- Marathon 동점 처리 규칙을 완전한 비교 순서로 고정하겠다는 회신은 적절하다.
- 첫 3판 개인 최고 기록 강화 연출을 기능 규칙으로 내리겠다는 회신은 적절하다.
- 광고/수익화 금지 구역을 설계 문서에 명시하겠다는 회신은 적절하다.
- 출석/연속 방문 구분, Daily Challenge 운영 가이드 보강 방향도 타당하다.

### 8.4 조건부 수용 항목

- 복귀 루프를 MVP blocker로 보지 않는 판단은 수용한다.
- 다만 이 항목을 단순 오픈 이슈로 남기면 다시 밀릴 가능성이 있으므로, `후속 설계 항목`으로 문서상 명시하고 최소한 방향성 선택지까지는 고정해야 한다.
- 공유 결과 페이지의 상위 퍼센트 강조는 MVP 필수는 아니지만, 경쟁 유도 효과가 높으므로 설계 보강 시 우선 검토 대상으로 유지하는 것이 좋다.

### 8.5 기획자 최종 요청

설계자 회신 이후에도 아래 3가지는 실제 설계문서 수정본에서 반드시 확인되어야 한다.

1. `06_MVP개발계획서.md`에 모바일/접근성/분석 최소 범위가 실제로 앞당겨 반영되었는지
2. `01_기능설계서.md`에 Marathon 동점 처리 규칙과 첫 3판 성취감 강화 규칙이 단일 해석 가능 문구로 들어갔는지
3. 광고 금지 구역과 복귀 UX 후속 방향이 문서상 명시되었는지

### 8.6 최신 기획자 결론

- 현재 상태: `설계자 회신 방향 수용`
- 최종 상태: `수정 반영본 확인 전까지 조건부 적합 유지`

- 복귀 사용자 경험은 후속 범위이지만, 더 이상 막연한 오픈 이슈가 아니라 선택 가능한 방향성 항목으로 문서상 구조화됐다.
- 출석/연속 방문 구분, Daily Challenge 난이도 운영 원칙, 공유 페이지의 비교 정보 우선 노출도 이전 검토 의견 대비 보강됐다.

- 현재 판정은 설계 문서 기준 `적합`이다.
- 이후 구현 단계에서는 문서상 합의된 규칙이 실제 화면, API, QA 기준에 동일하게 반영되는지 확인이 필요하다.
- 특히 모바일 HUD, 결과 화면 성취감 연출, 광고 금지 구역은 구현 과정에서 가장 쉽게 약화될 수 있으므로 계속 추적해야 한다.
